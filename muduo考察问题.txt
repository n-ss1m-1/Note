问题1：EventLoop 的核心机制

Q: EventLoop 如何实现"one loop per thread"模型？wakeupFd 的作用是什么？为什么需要 wakeup 机制？
考察点：
线程模型设计   跨线程唤醒机制   竞态条件避免

//
创建loop时记录相应的thread，然后通过isInLoopThread()判断当前运行的线程是否是属于该loop的，从而实现one loop per thread。
wakeupFd是用于在有任务到来的时候，解除loop阻塞，唤醒相应thread的机制。通过loop内的wakeupChannel关注可读事件，当别的线程调用queueInloop()递交任务时，通过wakeup()向wakeupFd写8字节数据，从而解除loop阻塞，实现快速的任务处理。
如果没有wakeup机制，则只有在loop所属的其他Channel有事件发生时，loop()才会解除阻塞，从而在handleEvent()之后才处理任务队列，这可能导致任务队列内的任务迟迟没有处理，导致服务器效率低下。
//
1）通过以下三种方式确保one loop per thread：1.声明：thread_local EventLoop* t_ loopInThisThread作为线程局部的保存实例。2.构造时检查：如果该声明不为空，则说明该loop已存在所属thread，则报错。3.运行时检查：isInLoopThread()比较loop创建时的thread和当前运行的thread的tid，保证thread一致。
2）wakeupFd用于线程间唤醒。通过注册wakeupChannel到loop所属的poller，并关注可读事件，当其他线程通过queueInLoop递交任务时，调用wakeup()向wakeupFd写8字节数据，触发可读事件，然后handleRead清空计数器，同时loop解除阻塞，可以快速的处理任务。也可以通过其他方式进行线程间通信，比如pipe。
3）wakeup机制：解决epoll_wait阻塞时的任务处理延迟。当loop阻塞在poll()上时，若其他线程向队列添加任务，如果没有wakeup机制，则loop继续阻塞，直到有网络事件发生才处理任务，这将导致延迟。wakeup机制保证任务队列处理的及时性，同时保证跨线程的任务递交的安全性。

问题3：智能指针的生命周期管理

Q: 在 TcpConnection 中，为什么使用 shared_from_this() 而不是传递 this 指针？在 handleClose 中，为什么要创建一个临时的 TcpConnectionPtr？
//
1. 为什么使用 shared_from_this() 而不是 this？
原因：确保异步回调期间对象存活
机制：
  1. 获取当前对象的 shared_ptr，增加引用计数
  2. 回调执行期间，对象不会被析构
  3. 回调结束，临时 shared_ptr 析构，引用计数减少

关键点：
  - 必须公有继承 std::enable_shared_from_this<TcpConnection>
  - 对象必须由 shared_ptr 管理
  - 避免悬垂指针和段错误

2. 传递 this 的风险
具体风险场景：
  时间线：
    t0: 线程A准备执行回调，持有 this
    t1: 线程B关闭连接，引用计数降为0
    t2: TcpConnection 被析构
    t3: 线程A执行回调，访问已销毁对象 → ❌崩溃

根本原因：
  - 没有增加引用计数
  - 无法阻止对象在回调执行期间被销毁
  - 多线程竞态条件

3. handleClose 中的临时 TcpConnectionPtr
目的：防止回调链中的自销毁
代码：
  void handleClose() {
      TcpConnectionPtr guard(shared_from_this());  // 引用计数+1
      connectionCallback_(guard);  // 安全
      closeCallback_(guard);       // 可能删除对象
  }  // guard 析构，如果引用计数为0，对象在此销毁

关键点：
  - 确保整个 handleClose 执行期间对象存活
  - 即使回调中删除对象，也会延迟到函数结束
  - 避免访问已销毁对象的成员变量

0         readerIndex_         writerIndex_        buffer_.size()

问题5：TcpConnection 的状态机

Q: TcpConnection 有哪几个状态？状态转换的触发条件是什么？为什么需要 kDisconnecting 状态？

考察点：

状态机设计

连接生命周期

优雅关闭


//
完整的状态转换图：

     构造函数
kDisconnected ──────> kConnecting
                                  		│
                                  		│ connectEstablished()
                                	 	↓
                             	  	kConnected
                                  		↓
        ┌────────────────────────┐
        │ 对端关闭/本端错误                              │ shutdown()
        ↓                                                	      ↓
   kDisconnected         			   kDisconnecting
        ↑                                                  	      │
        │ handleClose()                                     │ 发送完成
        │                                               	      ↓
        └────────────────────────> kDisconnected
                                                       			(handleClose)

详细触发条件：

1. kDisconnected → kConnecting：
   - TcpConnection构造函数中：state_ = kConnecting

2. kConnecting → kConnected：
   - connectEstablished()中：setState(kConnected)
   - 通常在IO线程中执行

3. kConnected → kDisconnecting：
   - 用户调用shutdown()时
   - 设置state_ = kDisconnecting
   - 但连接并未立即关闭

4. kDisconnecting → kDisconnected：
   - 发送缓冲区数据全部发送完成
   - handleWrite()中检查并调用shutdownInLoop()
   - 最终handleClose() → kDisconnected

5. kConnected → kDisconnected（异常路径）：
   - handleRead()读取到0字节（对端关闭）
   - handleError()发生错误
   - 都调用handleClose() → kDisconnected

//
1.四个状态，KDisconnected(已断开连接),KConnecting(正在连接),KConnected(已连接),KDisconnecting(正在断开连接/半关闭)
2.1)初始TcpServer收到连接请求之后，轮询ioloop，创建TcpConnection来管理连接，TcpConnection构造时初始化state_=KConnecting->TcpServer之后将TcpConnection加入连接映射表，设置回调，然后将TcpConnection::connectEstablished()加入其ioloop的任务队列，在该函数中设置Channel并setState(KConnected)->之后在连接过程中保持该状态。
2.1)如果handleRead收到对端0字节数据，表示客户端断开，则此时调用handleClose()，在handleClose中setState(KDisconnected),并回调connectionCallback_通知应用层，回调closeCallback_将该连接移除server的映射表并将connectDestroyed加入其ioloop的任务队列，将相应channel从poller中移除，最后析构TcpConnection。
2.2)如果在KConnected状态下，应用层调用TcpConnection::shutdown()，则TcpConnection将进入KDisconnecting状态，等待该连接的服务器端发送完数据之后，该连接的服务器端关闭写端，进入半关闭状态，等待客户端发送完所有数据之后，该连接将完全关闭。
3.kDisconnecting状态表示服务端连接的半关闭状态，如果进入KDisconnecting状态时，服务器还在发送数据，则会继续发送直到发送完毕之后才会关闭写端，这需要KDisconnecting状态的标记作用。同时该状态可以实现在接收完客户端的数据之后实现优雅关闭。


问题6：TcpServer 的线程模型

Q: 当新连接到达时，TcpServer 如何选择 EventLoop 来处理这个连接？这样做有什么好处？在多线程环境下，如何安全地将连接从主线程转移到子线程？

考察点：

多线程设计

负载均衡

线程间通信
//
1.
TcpServer采用主从Reactor模型：
主Reactor（main EventLoop）：运行在主线程，负责接受新连接
从Reactor（sub EventLoop）：运行在IO线程池，负责处理已建立连接的IO事件
此处通过threadPool_->getNextLoop()，轮询获取subloop来管理连接
2.通过给每个subloop均匀分配连接管理，实现负载均衡|+资源隔离(一个连接的处理不会阻塞其他连接)、性能优化(减少锁竞争)
3.getNextLoop()获取子线程ioLoop，然后在创建TcpConnection管理连接时，将ioLoop传给其构造函数进行初始化，后续调用回调函数、创建连接、销毁连接等大多通过queueInLoop或runInLoop交给其所属ioloop进行处理，从而将连接从主线程转移到子线程

问题7：sendInLoop 的智能发送策略

Q: 解释 sendInLoop 函数中的"智能发送策略"：为什么先尝试直接 write，如果失败再缓冲+事件监听？这种设计解决了什么问题？

考察点：

网络性能优化

阻塞/非阻塞处理

流量控制
//
直接write的优势：
1. 零拷贝：数据直接从用户空间到内核，无需经过应用层缓冲区
2. 低延迟：立即发送，无额外延迟
3. 简单高效：代码路径短，无额外开销
适用场景：
1. 发送缓冲区为空
2. 没有注册EPOLLOUT事件
3. 数据量较小，通常能一次发送

缓冲的原因->速度不匹配：应用层生成数据速度 > 内核发送数据速度:
1. TCP流量控制：接收方窗口限制
2. 网络拥塞控制：网络状况变化
3. 内核缓冲区已满：nwrote<0   errno=EAGAIN | EWOULDBLOCK
4. 接收方处理速度：接收方处理能力

缓冲+事件监听的优势：
1. 非阻塞：不会阻塞线程，CPU可以处理其他连接
2. 高效：只在真正可写时被唤醒
实现机制：
1. 将未发送数据存入outputBuffer_
2. 注册EPOLLOUT事件到poller
3. 当内核发送缓冲区有空闲时，epoll_wait返回
4. 调用handleWrite()继续发送


问题8：高水位回调机制

Q: 什么是高水位回调？它的作用是什么？在什么情况下会触发？

考察点：

流量控制

内存管理

背压机制
//
定义：
高水位回调（HighWaterMarkCallback）是TcpConnection中的一种流量控制机制，当发送缓冲区（outputBuffer_）中未发送数据的大小超过预设阈值时，用于通知应用层降低数据发送速率，防止内存无限增长。

作用：
1. 内存保护：
   - 防止因网络拥塞或对端处理慢导致发送缓冲区无限增长
   - 避免单个连接占用过多内存，影响系统稳定性

2. 流量控制（背压机制，Back Pressure）：
   - 通知应用层降低数据生产速率
   - 实现生产者-消费者模型的速率匹配

触发条件：
1. 发送缓冲区大小（outputBuffer_.readableBytes()）从低于阈值变为高于阈值
2. 在sendInLoop()中添加新数据到缓冲区时检查
3. 只会触发一次，直到缓冲区降到阈值以下再超过时再次触发

Thread::start()流程：
1. Thread::start()中初始化信号量sem(值为0)
2. 创建std::thread对象，子线程开始执行lambda
3. sem_wait(&sem)阻塞，等待子线程

//EventLoopThread::startLoop()为什么用条件变量而不用信号量？
1. 功能需求不同：
   - 信号量：简单的计数器，适合"等待一次"的场景
   - 条件变量：可检查任意条件，适合"等待条件成立"的场景

2. 在EventLoopThread中：
   - 需要等待的条件是：loop_指针被设置
   - 条件变量配合mutex可以安全地检查这个条件
   - 条件不满足时，wait()会释放锁，允许子线程设置loop_

3. 避免虚假唤醒：
   - 条件变量可能被意外唤醒（spurious wakeup）
   - 谓词确保即使被唤醒，也要检查条件是否真正满足
   
信号量也可以实现，但条件变量更符合"等待条件"的语义。

为什么EventLoop在子线程栈上创建？

1. One Loop Per Thread原则：
   - 每个IO线程有自己独立的EventLoop
   - EventLoop只在其所属线程中被访问
   - 避免跨线程访问的锁开销

2. 生命周期管理：
   - 栈对象：自动管理，子线程结束时自动销毁
   - 主线程创建传递：需要手动管理，容易出错
   - 堆对象：需要delete，可能泄漏

3. 线程局部性：
   - EventLoop包含线程特有数据（如epoll fd、任务队列）
   - 在所属线程栈上创建，缓存局部性更好
   - 符合"资源在哪使用，就在哪创建"的原则
