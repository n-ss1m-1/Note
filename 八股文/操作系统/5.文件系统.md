# 文件系统
#### 文件系统的基本组成
###### 核心定义
- 文件系统是操作系统中**管理持久化数据**的子系统，负责将用户数据存储到磁盘(断电后数据不丢失)，并对文件/目录进行组织管理
- Linux遵循**一切皆文件**的理念：普通文件、目录、设备、管道、socket都由文件系统同一管理

###### 核心数据结构
- **索引节点**(`inode`)：
	- **存储文件的元数据**：inode编号、文件大小、访问权限、时间戳、数据块指针等
	- 是**文件的唯一标识**，与文件一一对应，**持久化存储在磁盘中**
	- 但**不存储文件名**，也**不存储文件实际内容**

- **目录项**(`dentry`)：
	- 存储文件名、inode指针、目录层级关系
	- 作用：**将人类看得懂的文件名->翻译成操作系统看得懂的inode**
	- 由**内核维护**，**缓存在内存中**，不持久化到磁盘，用于加速目录查询

- **inode、dentry、目录 的关系**：
	- inode和dentry：目录项和inode是**多对一**的关系(一个文件可对应多个别名，如硬链接)
	- 目录和dentry：**目录**本身也是**文件**，其内容是子目录 / 文件的**目录项列表**

###### 磁盘存储分区
1. **超级块**：存储文件系统全局信息(块总数、块大小、空闲块数)，**文件系统挂载时加载进内存**
2. **索引节点区**：存储所有inode，**文件访问时加载进内存**
3. **数据块区**：存储文件 / 目录的实际数据，读写最小单位为**逻辑块/数据块**(Linux中通常为4KB)

#### 虚拟文件系统(VFS)
###### 核心作用
- 为上层提供统一的文件操作接口，屏蔽不同底层文件系统的差异，让开发者无需关系具体文件系统实现
###### 架构分层(Linux)
- 用户空间：应用程序->库函数->系统调用
- 内核空间：
	- 系统调用层：接收用户请求
	- **VFS层**：抽象接口层，定义通用数据结构与操作(open / read / write)
	- 具体文件系统层：Ext2/3/4、XFS、NFS等
	- 缓存层：页缓存、inode缓存、dentry缓存
	- 设备层：本地磁盘、网络存储
	![[Linux文件系统架构.PNG|340]]
###### 文件系统分类
- **磁盘文件系统**：数据持久化在磁盘，如Ext2/3/4、XFS
- **内存文件系统**：数据存储在内存，如/proc、/sys，用于读取内核数据
- **网络文件系统**：访问远程主机数据，如NFS、SMB
###### 关键特性
- 文件系统必须先**挂载**(mount)到某个目录才能使用，VFS负责管理多文件系统的挂载与访问

#### 空闲空间管理
- 用于管理磁盘中未被分配的空闲块，高效分配 / 回收磁盘空间
###### 空闲表法
- 原理：为所有空闲区域建立**表**，记录每个空闲区的**起始块号+块个数**(连续分配)
- 优点：适合连续空间分配，查找连续空间块较快
- 缺点：空闲区过多时，表体积庞大，查询效率低
###### 空闲链表法
- 原理：把**所有空闲块串联成链表**，每个空闲块包含指向下一个空闲块的指针
- 优点：实现简单，无需额外存储大表
- 缺点：随机访问效率低，链表操作需大量I/O，不适用于大型文件系统
###### 位图法(Bitmaps, Linux主流)
- 原理：用**二进制位表示磁盘块的状态**：`0`表示空闲，`1`表示已分配
- 优点：查找高效、空间开销小，同时支持数据块和inode的空闲管理
- 实现：将位图按块组划分，每个块组维护字节的位图，避免单张位图过大

#### 文件系统的结构(Linux Ext2)
###### 整体布局
- 磁盘由大量**块组**(Block Group)组成，每个块组结构一致，包含(除引导块)：
	- **引导块**：系统启动时用于引导
	- **超级块**：文件系统全局信息(冗余存储在多个块组，防止崩溃丢失)
	- **块组描述符**：记录块组内空闲块、inode数量等状态
	- **数据位图**：管理块组内数据块的空闲 / 使用状态
	- **inode位图**：管理块组内inode的空闲 / 使用状态
	- **inode列表**：存储块组内所有inode
	- **数据块**：存储文件 / 目录数据
- ![[块组.PNG|532]]
###### 设计思想
- **冗余备份**：超级块、块组描述符进行冗余存储，提升可靠性
- **局部性优化**：将文件数据和inode放在同一块组，减少磁头寻道时间，提升I/O性能
- **扩展优化**：Ext4等后续版本采用稀疏技术，避免冗余存储全局信息，进一步提升效率



# 目录
#### 目录的存储
###### 目录的本质
- Linux遵循一切皆文件，**目录本身也是文件**，也有自己的inode
- 普通文件的块存储**数据**，目录文件的块存储**目录项列表**
###### 目录项内容
每个目录项包含：
- 文件名
- 对应文件的inode号
- 文件类型
目录的特殊项：
- `.` ：当前目录，指向自身inode
- `..` ：上级目录，指向父目录inode
###### 存储格式
- **列表格式**：简单直接，逐个遍历查找，适合小目录
- **哈希表格式**(Ext文件系统)：对文件名做哈希运算，O(1)查找，适合大目录，但要处理哈希冲突
###### 性能优化
- 目录查询造成频繁磁盘I/O，内核会把**目录项缓存到内存**，后续访问直接读内存，减少I/O开销


#### 软链接和硬链接
##### 硬链接(Hard Link)
###### 核心原理
- 多个目录项指向**同一个inode**，共享文件数据
- 本质时**给文件起别名**，不占用额外磁盘空间(只新增目录项，不新增inode / 数据块)
###### 关键特性
- inode链接计数`i_links_count`会随硬链接数量增加
- 只有当**所有硬链接+源文件被删除**，且无进程打开该文件时，才会真正释放inode和数据块
- **不能跨文件系统**(inode编号仅在单个文件系统中唯一)
- 不能链接目录(避免目录树生成循环，破坏遍历逻辑)
![[硬链接.PNG|509]]

##### 软链接(Symbolic Link / 符号链接)
###### 核心原理
- 创建了**新文件**，有独立的inode和数据块
- 数据块里存的是**目标文件的路径字符串**，而非真实数据
- 相当于**快捷方式**，是**内核级的路径解析+跳转**
###### 关键特性
- 与源文件是两个独立文件，有各自的inode
- 可以**跨文件系统**，甚至链接不存在的文件
- 源文件删除后，软链接变成断链，访问时提示文件不存在
- 软链接本身有自己的权限、大小等元数据
![[软链接.PNG|505]]

##### 硬链接 vs 软链接

| 特性       | 硬链接         | 软链接                 |
| -------- | ----------- | ------------------- |
| inode 关系 | 共享同一个 inode | 独立 inode，与源文件不同     |
| 跨文件系统    | 不可以         | 可以                  |
| 链接目录     | 不可以         | 可以                  |
| 源文件删除后   | 链接仍有效，数据保留  | 链接失效（断链）            |
| 额外空间     | 仅新增目录项      | 新增 inode + 数据块（存路径） |
| 本质       | 文件别名        | 路径解析+跳转             |



# 文件
#### 文件的使用
###### 用户视角
用户通过**系统调用**操作文件：
`open()`->`read()/write()`->`close()`
![[用户使用文件系统的方式.PNG|522]]
###### 系统视角
- **每个进程**维护一张**打开文件表**，表项对应文件描述符
	- 文件指针：当前读写位置(进程唯一)
	- 引用计数：记录打开该文件的进程数，计数为0时释放表项
	- 文件磁盘位置、访问权限等状态信息
- 用户以**字节为单位**进行读写，操作系统以**数据块为单位**进行读写，文件系统负责屏蔽这层差异
- 文件系统的基本操作单位/最小寻址单位：**数据块**

#### 文件的存储
##### 连续空间存放方式
- 原理：文件数据存放在**连续的物理磁盘块**中，文件头(相当于inode)记录`起始块号+长度`
- 优点：顺序 / 随机读写效率极高(一次寻道即可读完整个文件)
- 缺点：
	- 产生外部碎片：删除文件后留下的小空闲块无法容纳大文件
	- 文件长度不易扩展：扩展时需挪动大量数据，效率极低
	![[连续空间存放方式.PNG|310]]

##### 非连续空间存放方式
###### 链表方式
1. **隐式链表**：
	- 原理：文件头记录`第一块+最后一块`的位置，每个**数据块内部存放**指向下一块的指针
	- 优点：无外部碎片，支持动态扩展
	- 缺点：
		- 只能顺序访问，无法随机访问
		- 指针占用额外空间
		- 指针损害会导致数据丢失
	- ![[隐式链表.PNG|439]]
2. **显式链表**(FAT)：
	- 原理：把所有块的指针**集中存放**在内存的**文件分配表**(**FAT**)中，文件头仅记录第一块号
	- 优点：支持随机访问，减少磁盘I/O
	- 缺点：表体积随磁盘容量增大而急剧膨胀，不适合大磁盘
	- ![[显式链表.PNG|155]]

###### 索引方式
- 原理：为每个文件创建**索引数据块**，存放指向**所有数据块的指针列表**，文件头指向该索引块
- 优点：
	- 无外部碎片，支持动态扩展
	- 支持顺序读写和随机访问(索引定位数据块)
- 缺点：小文件也需要额外分配索引块，存在存储开销
- 大文件的优化方式：
	- **链式索引块**：
		- 索引块之间使用**指针串联**(索引块内存放**指向下一个索引块**的指针)
	- **多级索引块**：
		- 高级索引块**存放多个低级索引块**，层层嵌套
		- 一级索引块存放指向数据块的指针

#### Unix文件的存储方式
![[早期Linux文件存储方式.PNG|391]]
- inode指针设计：
	- 10 个**直接指针**：指向数据块(适合小文件，<10 块)
	- 1 个**一级索引数据块**：指向索引块，索引块再指向数据块(支持中等文件)
	- 1 个**二级索引数据块**：指向索引块，索引块再指向一级索引块
	- 1 个**三级索引数据块**：指向索引块，索引块再指向二级索引块(支持超大文件)
- 优势：
	- 灵活适配不同大小的文件：小文件用直接指针，大文件用多级索引
	- 兼顾存储效率和访问性能

- 局限(Ext2/3)：
	- 大文件访问需要多次查询索引块，I/O 次数多，效率较低
	- Ext4 对此做了优化(如 extent 机制)，提升大文件访问效率

#### 存储方式对比

|存储方式|访问磁盘次数|优点|缺点|
|---|---|---|---|
|**连续分配**|1 次|读写速度极快，适合大文件|外部碎片，不易扩展|
|**隐式链表**|n 次（n 为块数）|无碎片，动态扩展|仅顺序访问，指针易损坏|
|**显式链表（FAT）**|内存查表|支持随机访问|表体积大，不适合大磁盘|
|**索引分配**|m 级索引需 m+1 次|支持随机访问，动态增删|小文件索引开销大|



#### 文件I/O
##### 缓冲I/O vs 非缓冲I/O
- 区别：**是否利用标准库缓存**

|类型|核心逻辑|优点|缺点|典型场景|
|---|---|---|---|---|
|**缓冲 I/O**|数据先存入**标准库缓冲区**，再通过系统调用访问文件|减少系统调用次数，降低 CPU 上下文切换开销|数据可能延迟输出（如行缓冲）|普通文件读写（`printf`/`fwrite`）|
|**非缓冲 I/O**|直接通过系统调用访问文件，不经过标准库缓存|数据实时性高，无延迟|系统调用频繁，开销大|日志、实时数据输出（`write`）|

##### 直接I/O vs 非直接I/O
- 区别：**是否利用内核页缓存**

|类型|核心逻辑|优点|缺点|典型场景|
|---|---|---|---|---|
|**直接 I/O** (`O_DIRECT`)|绕过内核缓存，直接在用户空间与磁盘间传输数据|无数据拷贝开销，适合精准控制 I/O|性能依赖磁盘，无缓存加速|数据库、自缓存应用|
|**非直接 I/O**（默认）|数据先拷贝到**内核页缓存**，再由内核决定何时刷盘|减少磁盘 I/O 次数，提升读写性能|存在两次数据拷贝（用户态 ↔ 内核态）|绝大多数普通文件|
- **内核缓存刷盘触发时机**：
	- 调用 `sync()`/`fsync()` 主动刷盘
	- 缓存数据量过大、内存紧张时自动刷盘
	- 缓存数据超时后自动刷盘

##### 阻塞/非阻塞 I/O  vs  同步/异步 I/O

**举例：饭堂做饭+阿姨打饭**
###### 核心前提：I/O 分为两个阶段
1. **数据准备阶段**：内核等待磁盘 / 网络数据就绪
2. **数据拷贝阶段**：内核将数据拷贝到用户进程缓冲区
###### 阻塞I/O
流程：进程发起`read`后，**一直阻塞等待**，直到数据准备+数据拷贝全部完成才返回
优点：逻辑简单，变成容易
缺点：CPU利用率低，等待期间进程无法做其他事情
![[阻塞IO.PNG|280]]

###### 非阻塞I/O
流程：
- 进程发起`read`后，若数据未就绪，**立即返回`EWOULDBLOCK`**，可继续指向其他任务
- 但是需要**不断轮询内核**，直到数据就绪后再次调用`read`完成拷贝
优点：不阻塞进程，可并发处理其他任务
缺点：
- 轮询会消耗CPU
- 数据拷贝阶段仍需同步等待
![[非阻塞IO.PNG|294]]

###### I/O多路复用
流程：
- 使用`select/poll/epoll`等系统调用，**同时监听多个文件描述符**的就绪状态
- 内核通知某个`fd`就绪后，进程再发起`read`完成数据拷贝
优点：单线程可高效管理多个I/O流，是高并发服务的核心
缺点：数据拷贝阶段仍需同步等待
![[IO多路复用.PNG|291]]

###### 异步I/O(AIO)
流程：
- 进程发起`aio_read`后**立即返回**，内核在后台完成 数据准备+数据拷贝工作
- 全部完成后，内核主动通知进程处理数据
优点：**全程无阻塞**，CPU利用率最高
缺点：编程复杂，API兼容性差
![[异步IO.PNG|282]]

##### 同步I/O vs 异步I/O

| 模型           | 数据准备阶段          | 数据拷贝阶段   | 本质     |
| ------------ | --------------- | -------- | ------ |
| **阻塞 I/O**   | 阻塞              | 阻塞       | 同步 I/O |
| **非阻塞 I/O**  | 不阻塞（轮询）         | 阻塞       | 同步 I/O |
| **I/O 多路复用** | 阻塞在 select/poll | 阻塞在 read | 同步 I/O |
| **异步 I/O**   | 不阻塞             | 不阻塞      | 异步 I/O |
**核心结论**：
- 只要「数据拷贝阶段」需要进程等待，就是**同步 I/O**
- 只有两个阶段都完全不阻塞，才是**异步 I/O**
