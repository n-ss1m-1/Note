# Linux是怎么收发网络包的？

#### 网络模型
###### OSI七层模型(理论参考)
- **应用层**：为应用程序提供统一接口
- **表示层**：数据格式转换与加密
- **会话层**：建立 / 管理 / 终止通信会话
- **传输层**：端到端数据传输(TCP/UDP)
- **网络层**：路由、转发、分片(IP)
- **数据链路层**：帧封装、MAC 寻址、差错检测
- **物理层**：物理介质传输比特流
###### TCP/IP四层模型(Linux实际应用)
- **应用层**：为应用程序提供统一接口(HTTP/DNS/FTP)
- **传输层**：端到端数据传输(TCP/UDP)
- **网络层**：路由、转发、分片(IP/ICMP)
- **网络接口层**：即数据链路层+物理层，帧封装、MAC 寻址、网卡传输

#### Linux接收网络包的流程
1. 网卡接收：网卡通过**DMA**将数据包放入**Ring Buffer**(环形缓冲区)
2. 通知内核+中断处理：
	- 传统方式：**硬中断**，**一个网络包->一个中断**，CPU频繁处理中断，性能下降
	- 高性能场景：**NAPI机制**(**软中断+轮询**)：中断处理函数暂时屏蔽中断，唤醒软中断轮询处理数据(从Ring Buffer中拷贝数据到struct sk_buff缓冲区)，**一次中断处理多个包**，提升CPU性能
3. 逐层解封装：
	- 网络接口层：去掉帧头和帧尾，判断上层协议(IPv4 / IPv6)，交给网络层
	- 网络层：去掉IP头，判断下一步(本机 / 转发)，交给传输层
	- 传输层：去掉TCP/UDP头部，根据四元组(源/目标端口、源/目标IP)找到对应socket，拷贝数据到对应socket的接收缓冲区
4. 应用读取：应用程序通过socket接口从接收缓冲区中读取数据

#### Linux发送网络包的流程
1. 应用调用：应用通过socket接口，将数据拷贝到socket发送缓冲区
2. 逐层封装：
	- 传输层：添加TCP/UDP头部，交给网络层
	- 网络层：添加IP头部，查询路由表，按MTU分片
	- 网络接口层：通过ARP获取下一跳MAC地址，添加帧头/帧尾，放入发包队列
3. 网卡发送：驱动通过DMA从发包队列读取数据包，放入网卡队列，由网卡硬件发送到网络
**关键优化**：
- NAPI机制：解决频繁硬中断导致的CPU开销，使用**软中断+轮询**批量处理数据包
- DMA技术：网卡直接与内存交互，避免CPU拷贝数据，提升吞吐量

# 零拷贝
#### 为什么需要DMA技术？
###### 无DMA时的I/O痛点
- CPU全程参与数据搬运：
	![[无DMA.PNG|544]]
- 问题：数据搬运期间，CPU完全阻塞，无法执行其他任务，大流量情况下CPU根本忙不过来

###### DMA技术的本质
- **DMA**(Direct Memory Access，直接内存访问)：让I/O设备(磁盘 / 网卡)直接与内核内存传输数据，CPU只负责发起命令和接收完成中断，不再参与「**I/O设备 ↔ 内核缓冲区**」这段搬运

###### DMA工作流程
![[有DMA.PNG|640]]



#### 传统的文件传输由多糟糕？
以 读取文件并通过网络发送 为例：
```
read(file, tmp_buf, len);
write(socket, tmp_buf, len);
```
![[传统文件传输.PNG|489]]

1. **上下文切换开销(4次)**
	- 2次系统调用(`read` + `write`)，每次系统调用都会触发：
		- 用户态->内核态
		- 内核态->用户态
	- 一共**4次上下文切换**，每次切换耗时几十纳秒到几微秒，高并发下会被严重放大
2. **数据拷贝开销(4次)**
	1. DMA：磁盘->内核缓冲区(`PageCache`)
	2. CPU：内核缓冲区->用户缓冲区
	3. CPU：用户缓冲区->Socket缓冲区
	4. DMA：Socket缓冲区->网卡
	- 问题：**文件传输**场景下，**用户态不会对数据做任何加工**，**内核->用户->内核**这两次CPU拷贝完全是**冗余**的，既浪费CPU资源，又降低了系统性能


#### 如何优化文件传输的性能？
1. **减少上下文切换**
	- 上下文切换由系统调用导致，所以要**减少系统调用次数**
	- 传统方式是2次系统调用，优化目标是1次系统调用完成整个传输流程
2. **减少数据拷贝**
	- 去掉内核缓冲区->用户缓冲区->Socket缓冲区的2次CPU拷贝，让数据直接在内核态流转
	- 最终目标：让数据只经过2次DMA拷贝(磁盘->内核->网卡)，**0次CPU拷贝**(零拷贝本质)



#### 如何实现零拷贝？
###### 1. `mmap() + write()`
- 原理：使用`mmap()`将内核缓冲区(`PageCache`)**映射到用户空间**，替代`read()`，**省去内核缓冲区->用户缓冲区的CPU拷贝**
- 流程：
	![[mmap.PNG|462]]
- 开销：**4次上下文切换(2次系统调用)+3次数据拷贝**
- 局限：仍有1次CPU拷贝，且上下文切换次数未减少

###### 2. `sendfile()`(Linux 2.1+)
- 原理：使用**1次**系统调用完成「读文件 + 发网络」，直接在内核态将内核缓冲区数据拷贝到Socket缓冲区，**完全不经过用户态**
- 流程：
	![[sendfile.PNG|462]]
- 开销：**2次上下文切换(1次系统调用)+3次数据拷贝**

###### 3. `sendfile()+SG-DMA`(Linux 2.4+)
- 原理：
	- 内核缓冲区描述符和数据长度传到Socket缓冲区后
	- **网卡的SG-DMA控制器**就可以直接**将内核缓冲区中的数据 ->拷贝到 网卡的缓冲区**
	- **省去「内核缓冲区 → Socket 缓冲区」的 CPU 拷贝**
- 流程：
	![[SG-DMA.PNG|481]]
- 开销：**2次上下文切换**(1次系统调用)+**2次数据拷贝**(2次均为DMA拷贝，**0次CPU拷贝**)
- **真正的零拷贝**：0次CPU拷贝，全程不需要CPU搬运数据，只通过DMA进行拷贝

###### DMA与零拷贝
- **DMA 解放了「I/O设备 ↔ 内核」的 CPU 搬运**
- **零拷贝进一步解放了「内核 ↔ 用户」的 CPU 搬运**

###### 零拷贝的前提
- **不允许用户态修改数据**：若需压缩 / 加密等操作，必须经过用户态，无法使用零拷贝
- **依赖PageCache**：零拷贝需要PageCache作为内核缓冲区(中间层)，大文件场景下失效

###### 对比不同的I/O方式

| 方式                    | 上下文切换次数 | 数据拷贝次数     | CPU 拷贝次数 | 适用场景        |
| --------------------- | ------- | ---------- | -------- | ----------- |
| 传统 I/O (`read+write`) | 4       | 4          | 2        | 小文件、简单场景    |
| `mmap+write`          | 4       | 3          | 1        | 需用户态操作数据的场景 |
| `sendfile`            | 2       | 3          | 1        | 纯文件传输场景     |
| `sendfile+SG-DMA`     | 2       | 2          | 0        | 高性能网络传输场景   |
| 异步 I/O + 直接 I/O       | 视实现而定   | 2 (直接 I/O) | 0        | 大文件、高并发场景   |


#### PageCache有什么作用？
###### 核心作用
- **缓存热点数据**：将最近访问的磁盘数据缓存在内存(`PageCache`)中，如果下次读取直接命中，则避免磁盘I/O
- **预读功能**：顺序读时提前加载后续数据块到PageCache中，利用局部性原理提升读写性能
###### 大文件场景的问题
- **缓存污染**：大文件会占满PageCache，导致热点小文件无法缓存，整体性能下降
- **缓存命中率低**：大文件部分数据再次访问的概率低，PageCache无法发挥作用，而且还多了一次DMA拷贝开销
- **针对大文件的传输，不应该使用PageCache**

#### 大文件传输用什么方式实现？
###### 大文件为什么不适合用零拷贝？
- 零拷贝依赖PageCache，大文件导致缓存污染和命中率下降，反而会降低性能
###### 异步I/O+直接I/O (`O_DIRECT`)
- **异步I/O**：将 I/O 拆分为「发起请求」和「处理完成」两部分，**进程发起请求后不阻塞等待**，可以继续执行其他任务，
- **直接I/O**：**绕过 `PageCache`**，直接读写磁盘，避免缓存污染
- 流程：
	![[异步IO+直接IO.PNG|444]]

###### 直接I/O的应用场景
1. **应用程序已经实现了磁盘数据的缓存**，此时不需要PageCache再次缓存
	- 如MySQL数据库，可以通过参数设置开启直接I/O
2. **传输大文件**，大文件会占满PageCache，导致热点小文件无法充分利用缓存(缓存污染)，而且大文件难以命中PageCache缓存，反而增加了DMA拷贝开销(命中率低)，此时适用直接I/O

**注**：直接I/O绕过了PageCache，就**无法享受** 如下内核对PageCache的优化：
- 内核I/O调度算法会缓存i你可能多的I/O请求在PageCache，最后**合并**成一个更大的I/O请求再发给磁盘，以减少磁盘的寻址操作
- 内核会**预读**后续I/O请求放到PageCache中，以减少对磁盘的I/O操作


# I/O多路复用：`select`/`poll`/`epoll`
#### 最基础的Socket模型
###### 核心流程
- **服务端**：`socket()` → `bind()` → `listen()` → `accept()` → `read()`/`write()` → `close()`
- **客户端**：`socket()` → `connect()` → `write()`/`read()` → `close()`
- **关键概念**：
    - **监听 Socket**：专门用于 `accept()` 等待新连接
    - **已连接 Socket**：与客户端完成三次握手后，用于数据传输
    - **两个队列**：半连接队列（`syn_recv`）、全连接队列（`established`）

###### `bind()` 的作用
- 绑定 **IP + 端口**：
    
    - 端口：内核通过 TCP 头的端口号找到对应应用程序
    - IP：指定监听哪个网卡的流量（一台机器可能有多个网卡）

###### Linux 一切皆文件
- Socket 在内核中以**文件描述符**形式存在，读写 Socket 和读写文件的 API 一致（`read`/`write`）。
- 每个 Socket 对应一个 `inode`，关联发送 / 接收队列（`sk_buff` 链表），用于高效处理网络数据包的分层封装 / 解封装。

#### 如何服务更多的用户？
###### 理论最大连接数
- TCP 连接由**四元组**唯一标识：`{本机IP, 本机端口, 对端IP, 对端端口}`
- 服务端端口固定，因此理论最大连接数 ≈ 客户端 IP 数 × 客户端端口数
- IPv4 下理论上限约为 **2^48**，但实际受限于：
    - 文件描述符上限（默认 1024，可通过 `ulimit` 修改）
    - 系统内存（每个连接占用内核数据结构）

#### 多进程模型
###### 核心思想
- 父进程监听新连接，`accept()` 后调用 `fork()` 创建子进程，子进程处理与客户端的读写。
- **子进程会复制父进程的文件描述符**，直接使用「已连接 Socket」通信。
- ![[多进程模型.PNG|472]]
###### 优点
- 实现简单，**进程间隔离性好**，一个进程崩溃不影响其他。
###### 缺点
- **资源开销大**：每个进程占用独立内存空间、文件描述符等
- **上下文切换重**：切换时需要保存 / 恢复虚拟内存、寄存器等，高并发下性能急剧下降
- **僵尸进程**：子进程退出后需父进程调用 `wait()`/`waitpid()` 回收资源，否则会占用系统内存

#### 多线程模型
###### 核心思想
- 线程是进程内的「轻量级执行流」，同进程内线程共享文件描述符、内存空间等资源，仅私有栈、寄存器等。
- 连接建立后，创建线程处理读写，上下文切换开销远小于进程。
###### 优化：线程池
- 避免频繁创建 / 销毁线程的开销，**提前创建若干线程**，将已连接 Socket 放入队列，由**线程池线程取出处理**。
- 需注意：**全局队列操作需加锁**，避免多线程竞争导致数据错乱。
- ![[多线程模型.PNG|634]]
###### 优点
- 资源开销比多进程小，上下文切换更快，能支持更多并发连接。
###### 本质问题
- 仍为「**一个连接对应一个线程**」，C10K 场景下仍需维护 1 万个线程，操作系统调度压力极大，无法解决根本问题。
- 线程阻塞在 I/O 上时(如 `read` 等待数据)，仍会浪费 CPU 资源。


#### I/O多路复用
##### 核心思想
- 目标：使用**单个线程/进程监听多个Socket**，只有当Socket有就绪事件(可读、可写)发生时才处理，避免为每个连接单独分配线程/进程
- 本质：**时间片复用**，一个线程/进程在时间维度上轮流处理多个连接的就绪事件，实现高并发

##### select/poll
###### 共同特点
- 存储结构：使用线性结构(数组/位图)存储待检测的Socket集合
- 流程：
	1. 用户态将所有待检测Socket**拷贝**到内核
	2. 内核**遍历**所有Socket，标记就绪事件
	3. 内核将整个集合**拷贝**回用户态
	4. 用户态再次**遍历**整个集合，找到就绪Socket处理
- 时间复杂度：O(n)->随着连接数增加，性能线性下降
- 本质缺陷：**两次遍历+两次拷贝**，高并发下开销巨大

##### `epoll`
###### 核心优化点
1. 高效数据结构：内核用**红黑树**(自平衡二叉查找树)管理待检测Socket，增删查时间复杂度O(log n)，**无需每次传入整个集合**，避免大量数据拷贝
2. 事件驱动机制：内核维护**就绪链表**，仅将有时间的Socket加入链表，`epoll_wait()`**拷贝返回就绪事件**，无需遍历所有Socket，时间复杂度O(k)（k为就绪事件数）
3. 触发模式灵活：支持**水平触发**(LT)和**边缘触发**(ET)

###### 两种触发模型

| 模式           | 含义                                | 特点                           | 适用场景                   |
| ------------ | --------------------------------- | ---------------------------- | ---------------------- |
| **水平触发（LT）** | 只要 Socket 有数据可读 / 可写，就会**持续通知**   | 安全，不会丢失事件；需多次 `epoll_wait()` | 默认模式，编程简单              |
| **边缘触发（ET）** | 仅在 Socket 状态变化时（如数据第一次到达）**通知一次** | 高效，减少系统调用次数；需一次性读完数据，否则会丢失事件 | 配合**非阻塞 I/O**使用，追求极致性能 |
**关键提示**：
- 边缘触发只在状态变化时通知一次，必须搭配**非阻塞 I/O**
- 如果使用阻塞I/O：
	- while()持续读：无数据时read()会阻塞线程，导致服务卡死
	- 只调用一次read()：如果一次性未读完，会导致数据丢失
- 如果使用非阻塞I/O：
	- 使用while()持续读：无数据时`read()`返回`EAGAIN`或`EWOULDBLOCK`
	- 此时安全退出read()循环


##### `select` vs `poll` vs `epoll`

| 特性    | select                   | poll       | epoll          |
| ----- | ------------------------ | ---------- | -------------- |
| 连接数限制 | 1024(默认)，`FD_SETSIZE` 限制 | 同 epoll    | 无(受系统 fd 上限限制) |
| 存储结构  | BitsMap(固定大小)            | 动态数组(链表形式) | 红黑树 + 就绪链表     |
| 时间复杂度 | O(n)                     | 同 select   | O(k)（k 为就绪事件数） |
| 数据拷贝  | 2 次(用户态→内核态→用户态)         | 同 select   | 0 次（仅传递事件）     |
| 触发模式  | 水平触发                     | 同 select   | 水平触发 + 边缘触发    |
| 性能    | 随连接数增加线性下降               | 同 select   | 高并发下性能稳定       |







# 高性能网络模式：Reactor和Proactor
#### 模式演进
1. **原始方案**：为每个客户端连接创建独立线程/进程->资源消耗大，无法支持高并发
2. **资源复用**：引入线程池，一个线程处理多个连接->但阻塞I/O会导致等待数据时无法处理其他连接
3. **非阻塞socket+轮询**：线程轮询调用非阻塞I/O判断是否有数据->但轮询要消耗CPU，而且连接数越多时，轮询效率越低
4. **I/O多路复用**：使用select/poll/epoll监听多个连接的就绪事件，只有当连接有数据时才唤醒线程->解决了轮询问题，提升了CPU利用率
5. **封装为模式**：基于I/O多路复用封装出Reactor模式，让开发者专注业务逻辑，无需关心底层网络细节
#### Reactor模式(非阻塞+同步)
###### 核心思想
- 别名：Dispatcher模式(事件分发模式)
- 核心角色：
	- **Reactor**：负责**监听I/O事件**(连接、可读、可写)，并将**事件分发**给对应的处理器
	- **处理资源池**：负责**处理事件**(read->业务逻辑->send)，单线程/多线程/多进程
###### 单Reactor+单进程/线程
- 结构：
	- 1个Reactor->监听事件和分发事件
	- 1个Acceptor->处理连接事件
	- 1个handler->回调处理可读、可写事件+业务处理
- ![[单Reactor+单线程.PNG|488]]
- 优点：实现简单，无进程间通信 / 数据竞争
- 缺点：
	- 无法利用多核CPU
	- 业务处理慢会阻塞整个进程，导致响应延迟
- 典型应用：Redis

###### 单Reactor+多进程/线程
- 结构：
	- 主线程：
		- 1个Reactor->监听所有事件+分发事件
		- 1个Acceptor->处理连接事件
		- 多个handler->回调处理可读、可写事件
	- 线程池(子线程)->业务处理
- ![[单Reactor+多线程.PNG|492]]
- 优点：
	- 充分利用多核CPU
	- 避免业务处理阻塞Reactor
- 缺点：
	- 单Reactor仍可能成为性能瓶颈
	- 多线程共享数据需要加锁
- 典型应用：早期 NIO 框架

###### 多Reactor+多进程/线程
- 结构：
	 - 主线程：
		- 1个MainReactor->监听连接事件+分发连接
		- 1个Acceptor->处理连接事件
	- 线程池(子线程)：
		- SubReactor：监听可读、可写、连接断开、错误等事件+分发事件
		- handler：**回调处理**SubReactor监听的事件
- ![[多Reactor+多线程.PNG|551]]
- 优点：
	- 多核CPU利用更充分
	- Reactor拆分避免成为单点瓶颈
	- 主线程/子线程分工明确
- 缺点：实现相对复杂
- 典型应用：Netty、Memcached(多线程)；Nginx(多进程)

#### Proactor模式(异步)
###### 核心思想
- 基于**异步I/O**实现，感知**已完成**的I/O事件(内核以完成数据读写并拷贝到用户空间)
- 流程：
1. `Proactor Initiator` 负责创建 `Proactor` 和 `Handler` 对象，并将 `Proactor` 和 `Handler` 都通过 `Asynchronous Operation Processor` 注册到内核
2. `Asynchronous Operation Processor` 负责处理注册请求，并处理 [[5.文件系统#异步I/O(AIO)|异步I/O]] 操作
	1. 应用进程发起异步I/O请求(如`aio_read`)，立即返回，继续执行其他任务
	2. 内核完成数据准备+将数据拷贝到用户空间后，主动通知应用进程
	3. 应用进程直接处理已就绪的数据，无需再主动调用`read/write`
3. `Asynchronous Operation Processor` 完成 异步I/O 操作后通知 `Proactor`
4. `Proactor` 根据不同的事件类型回调不同的 `Handler` 进⾏业务处理
5. `Handler` 完成业务处理
- ![[Proactor.PNG|590]]
###### 局限与应用
- Linux平台：异步I/O不完善，网络socket不支持真正的内核级异步I/O，aio系列函数仅支持本地文件，因此高性能网络程序仍以Reactor为主
- Windows平台：通过IOCP实现了完整的异步socket支持，适合使用Proactor模式


#### `Reactor` vs `Proactor`

| 维度          | Reactor 模式                | Proactor 模式         |
| ----------- | ------------------------- | ------------------- |
| I/O 模型      | 非阻塞同步 I/O                 | 异步 I/O              |
| 事件类型        | 「待完成」事件(数据可读 / 可写)        | 「已完成」事件(数据已拷贝到用户空间) |
| 数据拷贝        | 应用进程主动调用 `read/write` 完成  | 内核自动完成，应用进程直接处理     |
| 编程复杂度       | 较低，成熟方案多                  | 较高，依赖操作系统异步 I/O 支持  |
| Linux 适用性   | 高，主流方案（Redis/Nginx/Netty） | 低，网络场景无完善支持         |
| Windows 适用性 | 一般                        | 高，IOCP 完美适配         |
