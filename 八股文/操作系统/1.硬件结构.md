
# CPU缓存一致性
#### CPU Cache 的数据写入
- **层级结构**：L1/L2 是每个 CPU 核心独占，L3 是多核共享；Cache 由多个 `Cache Line` 组成，是 CPU 与内存交互的基本单位，结构为「Tag 标志 + Data Block 数据块」。
- **读写逻辑**：CPU 优先读写 Cache，未命中时从内存加载数据到 Cache；写入时存在两种策略：
    1. **写直达（Write Through）**：数据同时写入 Cache 和内存，保证数据一致，但每次写操作都要访问内存，性能较差。
    2. **写回（Write Back）**：数据先写入 Cache 并标记为「脏（Dirty）」，只有当该 Cache Line 被替换时才写回内存，大幅减少内存访问次数，性能更高，但会带来多核缓存不一致问题。

#### 多核缓存一致性问题
- **问题根源**：多核 CPU 中，每个核心有独立的 L1/L2 Cache，当多个核心操作共享变量时，若一个核心修改了数据（写回策略下仅更新 Cache），其他核心的 Cache 仍持有旧值，导致数据不一致，程序执行结果错误。
- **解决目标**：要实现缓存一致性，必须满足两点：
	1. **写传播（Write Propagation）**：一个核心的 Cache 更新事件，必须通知其他所有核心
    2. **事务串行化（Transaction Serialization）**：所有核心看到的数据修改顺序必须一致，避免因事件到达顺序不同导致的数据不一致。

#### 总线嗅探机制
- **实现原理**：基于总线广播，当一个核心更新 Cache 时，会通过总线向所有核心广播事件，其他核心监听总线（嗅探），若发现自己的 Cache 中存在对应数据，就更新或失效该数据。
- **局限性**：
    - 每次更新都要广播，会增加总线负载。
    - 只能保证「写传播」，无法保证「事务串行化」，需要更上层协议来解决。

#### MESI 协议
MESI 是基于总线嗅探实现的缓存一致性协议，通过定义 4 种 Cache Line 状态，同时满足「写传播」和「事务串行化」：
###### 四种状态定义

|状态|英文|含义|
|---|---|---|
|**M**|Modified|已修改：数据仅在本核心 Cache 中，且已被修改（脏数据），与内存不一致，替换时需写回内存。|
|**E**|Exclusive|独占：数据仅在本核心 Cache 中，且与内存一致，其他核心无此数据，可自由修改。|
|**S**|Shared|共享：数据在多个核心 Cache 中，且与内存一致，修改前需先通知其他核心失效。|
|**I**|Invalid|已失效：数据无效，不可读取，需从内存重新加载。|

###### 核心状态流转
- **读取场景**：
    - 从内存加载数据 → 状态变为 E（独占）。
    - 其他核心也读取同一份数据 → 状态变为 S（共享）。
- **写入场景**：
    - 状态为 E/M：直接修改，状态变为 M，无需广播。
    - 状态为 S：先广播请求，让其他核心将对应 Cache Line 置为 I（失效），再修改本核心数据，状态变为 M。
- **替换场景**：状态为 M 时，需先将数据写回内存，再替换 Cache Line。

###### 协议优势
- 仅在必要时广播（如 S 状态写入），减少总线带宽压力。
- 通过状态机约束，天然实现「事务串行化」，保证多核数据一致。

#### 总结
1. **写入策略**：写直达简单但性能差，写回性能高但会引发缓存一致性问题。
2. **一致性要求**：必须同时满足「**写传播**」和「**事务串行化**」。
3. **实现方案**：总线嗅探是基础机制，**MESI 协议**是工业界主流的缓存一致性解决方案，通过 M/E/S/I 四种状态的流转，高效保证多核 Cache 数据一致。


# CPU 是如何执行任务的？
#### CPU 读写数据与缓存机制
###### 为什么需要 CPU Cache？
- **速度差**：CPU 访问 L1 Cache 比内存快约 100 倍，Cache 作为 CPU 与内存的缓冲层，能大幅减少对内存的访问频率，提升性能。
- **存储金字塔**：从寄存器 → L1/L2/L3 Cache → 内存 → 磁盘，容量越来越大，但访问速度越来越慢、成本越来越低。

###### CPU 读写数据的基本单位
- **Cache Line（缓存行）**：CPU 从内存读取数据的最小单位，通常为 **64 字节**，一次会加载一整块连续内存到 Cache 中。

###### Cache 伪共享问题
- **本质**：多个 CPU 核心的线程修改同一条 Cache Line 里的不同变量时，会导致 Cache 频繁失效、重新加载，严重影响性能。
- **发生过程**（以双核心为例）：
    1. 两个线程分别加载同一条 Cache Line 里的变量 A和B，Cache Line 变为「共享」状态。
    2. 核心 1 修改 A，会通知核心 2 将对应 Cache Line 标记为「已失效」。
    3. 核心 2 修改 B 时，发现 Cache 失效，需从内存重新加载，循环往复导致性能损耗。

###### 避免伪共享的方法
- **内存对齐填充**：用宏（如 Linux 内核 `__cacheline_aligned_in_smp`）让变量按 Cache Line 大小对齐，使变量分布在不同 Cache Line。
- **字节填充 + 继承**（如 Disruptor 框架）：在共享变量前后填充无用的 long 类型变量，占满整个 Cache Line，避免共享变量被加载到同一条 Cache Line。
- **核心思想**：用**空间换时间**，牺牲部分 Cache 空间来避免 Cache 失效，换取性能提升。

#### CPU 线程调度机制
###### 进程 vs 线程（Linux 视角）
- Linux 内核中，进程和线程都用 `task_struct` 表示，线程是「轻量级进程」，共享进程的大部分资源（内存、文件描述符等）。
- 调度器的调度对象是 `task_struct`（统称为「任务」）。

###### 调度类

|调度类|调度器|调度策略|适用场景|优先级|
|---|---|---|---|---|
|Deadline|Deadline 调度器|`SCHED_DEADLINE`|硬实时任务|最高|
|Realtime|RT 调度器|`SCHED_FIFO`/`SCHED_RR`|软实时任务|次之|
|Fair|CFS 调度器|`SCHED_NORMAL`/`SCHED_BATCH`|普通任务|最低|

- **实时任务**：优先级 0~99，对响应时间要求极高
- **普通任务**：优先级 100~139，侧重公平性。
- 调度优先级：`Deadline > Realtime > Fair`，实时任务会优先于普通任务执行。

###### 完全公平调度（CFS）
- **核心思想**：为每个任务分配虚拟运行时间 `vruntime`，运行越久的任务 `vruntime` 越大，调度器优先选择 `vruntime` 最小的任务执行，保证公平性。
- **计算公式**：`vruntime += 实际运行时间 * NICE_0_LOAD / 权重`
- 权重由 `nice` 值决定：`nice` 值越小 → 权重越大 → `vruntime` 增长越慢 → 优先级越高。

###### CPU运行队列
- **运行队列（Run Queue）**：每个 CPU 核心有独立的运行队列，包含 `dl_rq`（Deadline）、`rt_rq`（Realtime）、`cfs_rq`（Fair）三类，`cfs_rq` 用红黑树按 `vruntime` 排序。

###### 优先级调整命令
- 启动任务时用 `nice -n <值> <命令>` 设置 nice 值（范围 -20~19，-20 优先级最高）。
- 运行中用 `renice <值> -p <pid>` 修改 nice 值。
- 若需更强实时性，可通过 `chrt` 将任务改为实时调度策略（如 `SCHED_FIFO`）。


# 软中断
###### 为什么会有软中断？
- **硬中断的痛点**：硬中断由硬件触发（如网卡收包、键盘输入），执行时会**临时关闭中断**。如果硬中断处理逻辑复杂、耗时过长，会导致后续新的中断无法响应，甚至**丢失中断请求**。
- **软中断的设计**：将中断处理拆分为**上半部(硬中断)和下半部(软中断)**，实现 “快速响应 + 延迟处理” 的分离模式。

###### 上半部 vs 下半部

| 维度       | 上半部（Hard IRQ）                                    | 下半部（Soft IRQ）                                   |
| :------- | :----------------------------------------------- | :---------------------------------------------- |
| **触发时机** | **硬件触发**（如网卡收到数据包）                               | **内核触发**（硬中断执行完后主动触发）                           |
| **执行特点** | **极快执行**、关中断、**只做最核心**的硬件操作（如把数据拷贝到内存、更新硬件寄存器状态） | **延迟执行**、开中断、**处理耗时复杂**的逻辑（如网络协议栈解析、数据整理）       |
| **类比生活** | 外卖员电话通知你 “货到了”（只通知，不废话）                          | 你挂了电话，下楼拿外卖并和配送员闲聊（处理后续琐事）                      |
| **执行载体** | 中断上下文（优先级极高）                                     | **软中断内核线程**（如 `ksoftirqd/0`，属于内核进程，优先级低于普通用户进程） |
**执行流程**：硬件触发硬中断 → 快速拷贝数据 → 触发软中断 → 内核线程 `ksoftirqd` 异步处理复杂逻辑

###### 常见软中断类型
- 可通过 `/proc/softirqs` 查看
- **NET_RX**：网络数据包**接收**中断（高并发 Web 服务器最常见的瓶颈点）。
- **NET_TX**：网络数据包**发送**中断。
- **TIMER**：定时器中断。
- **RCU**：读 - 复制 - 解锁中断（内核常用的同步机制）。
- **BLOCK**：块设备中断（如磁盘 I/O）。
- **SCHED**：调度器相关中断。

###### 如何定位软中断 CPU 使用率过高的问题？
1. **查看整体 CPU 使用率**（top）
	- 重点关注 **`si` 字段**(`soft irq`)：如果 `si` 数值很高，说明 CPU 时间大量消耗在软中断上
	- 查看 `ksoftirqd` 进程：如果该进程 CPU 占用高，证明软中断处理繁忙。
2. **查看软中断详细统计**（`/proc/softirqs`）
	- 查看累计次数：`cat /proc/softirqs`
	- 重点观察：**NET_RX** 的累计次数在各 CPU 间是否分布均匀？增长速率是否异常？
	- 动态监控变化：`watch -d cat /proc/softirqs`（查看数值变化速率，突变通常意味着异常流量）。
3. **定位网络流量根源**
	如果发现 `NET_RX` 软中断激增：
	1. 查看网卡流量：`sar -n DEV 1`
	2. 抓包分析来源：`tcpdump -i <网卡名>`
	3. 决策：
	    - **非法攻击 / 流量**：配置防火墙规则拦截。
	    - **正常业务洪峰**：考虑网卡 / 服务器硬件升级，或优化网络协议栈。
