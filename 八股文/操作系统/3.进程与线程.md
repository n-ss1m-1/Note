# 进程
#### 进程的本质
###### 程序 vs 进程
- 程序：硬盘上的**静态二进制文件**
- 进程：**运行中的程序**，被加载到内存，占用CPU执行指令，是**系统资源分配的基本单位**
###### 并发 vs 并行
- 并发：单核CPU上多个进程**交替运行**(时间片轮转)，宏观上同时进行，微观上串行
- 并行：多核CPU上多个进程**同时在不同核心运行**，真正的微观同时运行

###### ❓ 高频考题预测

1. **【基础】** 请用一句话区分程序和进程。
    - **静态文件** vs **动态执行**、**资源分配单位**

2. **【场景】** 单核 CPU 能实现并行吗？为什么？
    - **不能**、**微观串行**、**时间片轮转**

3. **【深入】** 为什么说“进程是资源分配的最小单位”，而线程是“调度的最小单位”？
    - **独立地址空间**、**系统资源**、**轻量级**、**共享内存**

###### ⚠️ 避坑指南

- ❌ **错误说法**：“进程就是正在运行的程序。”（太口语化，缺乏资源视角）
- ✅ **正确说法**：“进程是程序的一次执行过程，它不仅包含代码和数据，还拥有独立的地址空间和系统资源（如文件句柄、内存页表）。”
- ❌ **错误说法**：“单核CPU通过超线程技术实现了并行。”
- ✅ **正确说法**：“超线程（Hyper-Threading）是在单个物理核心上模拟出两个逻辑核心，本质仍是利用指令级并行提高利用率，但在操作系统层面，单核同一时刻只能执行一条指令流，仍是并发。”

###### 🚀 进阶加分项

- **提到资源隔离**：强调进程间内存隔离（虚拟地址空间不同），一个进程崩溃不会影响其他进程（对比线程）。
- **提到现代架构**：举例 Chrome 浏览器采用“多进程架构”（每个标签页一个进程），正是利用了进程的隔离性来保证稳定性。

#### 进程的状态与变迁
###### 基础三状态
- **运行态(Running)**：进程占用CPU执行指令
- **就绪态(Ready)**：进程已准备好运行，等待CPU调度
- **阻塞态(Blocked)**：进程等待某事件(I/O、信号)，暂时无法运行，即使CPU空闲也不能调度
###### 拓展状态
- **创建态(New)**：进程正在被创建，初始化资源
- **结束态(Exit)**：进程执行完毕，资源被回收
- 挂起态：进程空间从内存被换出到磁盘
	- **阻塞挂起**：在外存中等待事件
	- **就绪挂起**：在外存但可随时换入内存运行

2. **【辨析】** “就绪态”和“阻塞态”的进程都能被调度器选中吗？
    - **只有就绪态**、**阻塞态不可调度**、**需先唤醒**

3. **【场景】** 如果一个进程一直在“就绪态”却得不到执行，可能是什么原因？
    - **饥饿现象**、**优先级过低**、**调度算法不公平**

###### ⚠️ 避坑指南

- ❌ **致命错误**：“阻塞态的进程可以被调度。”
- ✅ **纠正**：**只有就绪态**的进程才有资格被调度器选中进入运行态。阻塞态必须等待事件发生先转为就绪态。
- ❌ **混淆概念**：“挂起就是阻塞。”
- ✅ **纠正**：挂起（Suspend）是指**内存交换**（换出到磁盘），阻塞（Block）是指**等待事件**。一个进程可以既是阻塞又是挂起（阻塞挂起）。

###### 🚀 进阶加分项

- **Linux 实战**：提到在 Linux 中使用 `ps -o state,pid,cmd` 可以看到进程状态代码：`R` (Running/Ready), `S` (Interruptible Sleep/Blocked), `D` (Uninterruptible Sleep, 通常指等待磁盘 I/O，不可被杀死)。
- **僵尸与孤儿**：补充说明“结束态”的特殊情况：
    - **僵尸进程 (Zombie)**：子进程结束了，但父进程没调用 `wait()` 回收，PCB 仍残留。
    - **孤儿进程 (Orphan)**：父进程先死了，子进程被 `init` (PID 1) 收养。


#### 进程控制块(PCB)
###### 定义
- PCB是**进程存在的唯一标识**，包含进程的所有管理信息，存储在**内核空间**
###### PCB具体包含什么信息？
- **进程描述信息**：PID(进程ID)、UID(用户ID)
- 控制与管理信息：当前状态、优先级、调度信息
- 资源分配清单：虚拟内存地址、打开文件、I/O设备
- **CPU上下文**：寄存器值、程序计数器
###### 组织方式
- **链表**：同状态进程使用链表串联(就绪队列、阻塞队列...)，便于插入/删除
- **索引**：按状态建立索引表，表项指向PCB

#### 🔥 面试备战包

###### ❓ 高频考题预测

1. **【定义】** 操作系统是如何“感知”到一个进程存在的？
    - **PCB**、**唯一标识**、**内核数据结构**

2. **【结构】** PCB 中哪部分信息在进程切换时最关键？为什么？
    - **CPU 上下文**、**寄存器**、**程序计数器**、**恢复现场**

3. **【设计】** 为什么 Linux 内核要把 PCB 设计成 `task_struct` 这么复杂的结构体？
    - **管理复杂性**、**资源多样性**、**调度策略**、**兼容性**

###### ⚠️ 避坑指南

- ❌ **错误理解**：“PCB 只存了 PID 和状态。”
- ✅ **纠正**：PCB 最核心的作用是**保存现场**（CPU 上下文）。如果没有保存寄存器和 PC 指针，进程切换后就无法从断点继续执行了。
- ❌ **忽略细节**：认为 PCB 存储在用户空间。
- ✅ **纠正**：PCB 是内核数据结构，存储在**内核空间**，用户程序无法直接访问。

###### 🚀 进阶加分项

- **源码级认知**：直接说出 “在 Linux 源码中，PCB 的具体实现就是 `struct task_struct` 结构体。”（面试官听到这个会眼前一亮）。
- **内存布局**：提到每个进程都有一个**内核栈**，用于在内核态执行时保存临时数据和函数调用链，它与 `task_struct` 紧密相关。


#### 进程的控制操作

|操作|核心流程|
|---|---|
|**创建进程**|分配 PID → 申请 PCB → 分配资源 → 初始化 PCB → 加入就绪队列|
|**终止进程**|查找 PCB → 终止子进程 → 回收资源 → 从队列删除 PCB|
|**阻塞进程**|保存 CPU 上下文 → 置为阻塞态 → 加入阻塞队列 → 调度新进程|
|**唤醒进程**|从阻塞队列移除 PCB → 置为就绪态 → 加入就绪队列 → 等待调度|

###### ❓ 高频考题预测

1. **【系统调用】** `fork()`, `exec()`, `wait()` 分别对应上述哪个操作？它们的作用是什么？ 
	- **创建**、**加载程序**、**回收/同步**、**父子进程**

2. **【原理】** `fork()` 创建子进程时，会立即复制父进程的所有内存吗？ 
	- **写时拷贝 COW**、**共享页面**、**只读标记**、**按需复制**

3. **【异常】** 父进程退出前没有回收子进程，会发生什么？如何避免？ 
	- **僵尸进程**、**资源泄露**、**`wait()`/`waitpid()`**、**信号处理**

###### ⚠️ 避坑指南

- ❌ **经典误区**：“`fork()` 会完整复制一份父进程的内存给子进程，所以很慢。”
- ✅ **纠正**：现代 OS 使用 **写时拷贝 (Copy-on-Write, COW)** 技术。`fork` 时只是共享内存页并标记为只读，只有当任意一方尝试**写入**时，才真正复制该页。这极大提高了创建速度。
- ❌ **逻辑漏洞**：认为子进程会自动继承父进程的所有资源（如打开的文件锁）。
- ✅ **纠正**：大部分资源（如内存、寄存器）不继承或重置，但**打开的文件描述符**默认是继承的（需注意关闭不需要的 fd）。

###### 🚀 进阶加分项

- **特殊调用**：提到 `vfork()`（不复制地址空间，子进程直接在父进程空间运行，直到调用 `exec` 或退出，风险高但极快）和 `clone()`（Linux 特有，可指定共享哪些资源，是线程库 `pthread` 的底层实现）。
- **Init 进程**：强调系统中第一个进程 `init` (PID 1) 的职责：它是所有孤儿进程的“养父”，负责回收它们的资源，防止僵尸进程堆积。

#### 进程上下文切换
###### 什么是上下文？
- **CPU上下文 = CPU寄存器信息(程序计数器、栈指针、通用寄存器、状态标志位)**  
- **进程上下文 = CPU上下文 + 进程属性(进程ID、状态、fd) + 地址空间(页表基址、TLB)
- **线程上下文 = CPU上下文（共享进程地址空间，无需额外切换）**
###### 切换流程
- 保存当前进程的进程上下文到其PCB
- 从下一个进程的PCB中恢复进程上下文到CPU
- 跳转到新进程的程序计数器位置，继续执行
###### 切换上下文的场景
- **调度**(**保存进程上下文**)：
	- 时间片耗尽
	- 进程阻塞/挂起
	- 高优先级进程抢占
- **中断**(**保存CPU上下文**，因为中断处理仍使用原进程的地址空间)：
	- 硬件中断触发

#### 🔥 面试备战包

###### ❓ 高频考题预测

1. **【性能】** 为什么频繁的上下文切换会导致系统性能下降？
    - **开销大**、**保存/恢复现场**、**TLB 失效**、**缓存命中率降低**

2. **【细节】** 进程切换和线程切换，谁的开销大？为什么？
    - **进程切换大**、**切换页表**、**TLB 刷新**、**线程共享地址空间**

3. **【排查】** 如果服务器负载很高但 CPU 使用率不高，可能是什么原因？
    - **频繁上下文切换**、**大量 I/O 等待**、**锁竞争**、**`vmstat` 监控**

###### ⚠️ 避坑指南

- ❌ **片面理解**：“上下文切换只是保存几个寄存器。”
- ✅ **纠正**：进程切换不仅保存寄存器，还要**切换页表（Page Table）**，这会导致 **TLB（页表缓存）失效**，后续内存访问需要重新查页表，开销巨大（可能消耗数千条指令周期）。
- ❌ **忽视场景**：认为只有调度器才会触发切换。
- ✅ **纠正**：**中断**（如网卡收包、磁盘 IO 完成）是触发上下文切换的常见隐形杀手。

###### 🚀 进阶加分项

- **监控工具**：提到可以使用 `vmstat 1` 命令查看 `cs` (context switch) 列，或者 `pidstat -w` 查看具体进程的切换次数。
- **优化策略**：
    - **减少切换**：使用线程代替进程（线程切换不用切页表）；
    - **绑定核心**：使用 `taskset` 将关键进程绑定到特定 CPU 核心（CPU Affinity），减少缓存失效；
    - **用户态调度**：提及 **协程 (Coroutine)** 技术，在用户态进行调度，完全避免内核态上下文切换的开销（如 Go 语言的 GMP 模型）。


# 线程
#### 线程的定义
1. 线程是进程中的一条**执行流程**
2. 同一进程内的多个线程**共享代码段、数据段、文件描述符**等资源
3. 每个线程拥有**独立的寄存器和栈**，保证控制流相对独立
注：
- **共享数据**：全局变量、静态变量、堆、进程地址空间
- **非共享数据**：寄存器、栈(局部变量)、线程局部存储(`__thread`申明)
#### 线程 vs 进程

| 对比维度     | 进程                            | 线程                          |
| -------- | ----------------------------- | --------------------------- |
| **资源分配** | 拥有**独立**虚拟地址空间、文件、I/O 等完整资源   | **共享进程资源**，**仅拥有独立寄存器、栈**   |
| **调度角色** | **资源分配的基本单位**                 | **CPU 调度的基本单位**             |
| **状态**   | 具有创建 / 就绪 / 运行 / 阻塞 / 结束等状态   | 同样具有就绪 / 阻塞 / 运行三种基本状态及转换关系 |
| **切换开销** | 大：**CPU上下文(寄存器信息+进程属性+地址空间)** | 小：共享地址空间，仅保存线程私有数据(线程上下文)   |
| **通信方式** | 需 IPC 机制（管道 / 消息队列 / 共享内存）    | 直接读写全局变量，无需内核介入             |
| **健壮性**  | **一个进程崩溃不影响其他进程**             | **一个线程崩溃会导致整个进程崩溃**         |
**线程的优点**：
- 并发1：一个进程内可同时存在多个线程
- 并发2：线程间可**并发**执行
- 共享：线程间**共享**进程的地址空间和文件资源，数据交换更高效
- 控制：创建 / 终止 / 切换**开销远小于**进程
**线程的缺点**：
- 一个线程崩溃会导致所属进程的**所有线程崩溃**
- 共享资源易引发**数据竞争**，需要同步机制保障线程安全

#### 线程的上下文切换
1. **同进程内线程切换**：共享虚拟地址空间，无需更换页表，仅需保存线程上下文(寄存器、栈)，开销极小
2. **跨进程的线程切换**：等价于进程上下文切换，需切换地址空间、页表、TLB等资源

#### 线程的实现方式
###### 三种实现模型
- **用户线程**(UT)：在**用户态实现**，由用户态线程库管理，**内核不可见**，**TCB存于用户空间**
- **内核线程**(KT)：在**内核实现**，由内核直接管理，TCB(线程控制块)**存于内核空间**
- **轻量级线程**(LWP)：内核支持的用户线程，与内核线程一一对应，是用户态与内核态的桥梁(Linux经典实现)
###### 用户线程 vs 内核线程
- **用户线程(N:1模型)**：
	- 多个用户线程对应一个内核线程，内核只感知进程
	- **优点**：切换在用户态完成，速度极快，创建数量灵活
	- **缺点**：一个线程阻塞会导致整个进程阻塞，无法利用多核CPU并行；时间片分配给进程，线程间公平性差
- ![[用户线程.PNG|379]]
- **内核线程(1:1模型)**：
	- 一个用户线程对应一个内核线程，内核直接调度每个线程
	- **优点**：一个线程阻塞不影响其他线程，可利用多核CPU并行
	- **缺点**：创建 / 切换需系统调用，开销大，内核线程数量有限
- ![[内核线程.PNG|378]]

###### LWP与映射模型
- **1:1模型**：一个用户线程对应一个LWP->一个内核线程
	- 优点：实现并行，阻塞不影响其他线程
	- 缺点：创建开销大
- **N:1模型**：多个用户线程对应一个LWP->一个内核线程
	- 优点：切换高效，用户态管理
	- 缺点：一个线程阻塞则整个进程阻塞，无法利用多核CPU并行
- **M:N模型**(M>N)：多个用户线程对应多个LWP->多个内核线程
	- 优点：总和前两者优势，大部分切换在用户态，可利用多核CPU并行
	- 缺点：实现复杂，调试难度高
- **组合模式**：混合1:1与M:N模型，根据场景调整内核线程数量，平衡性能与开销

# 调度
#### 定义
- **调度**(scheduler)：操作系统从就绪队列中选择一个进程分配给CPU执行的过程，是实现多进程并发的核心机制

#### 调度时机
- 创建态->运行态：新进程创建后进入就绪队列，调度器选择一个进程运行
- 运行态->阻塞态：进程因I/O、等待事件阻塞，CPU空闲，需要调度新进程
- 运行态->结束态：进程退出，调度器从就绪队列选择下一个进程运行
- 运行态->就绪态：时间片用完/高优先级进程抢占，当前进程被挂起

#### 调度原则
- **CPU利用率**：让CPU尽可能忙碌，避免空闲
- **系统吞吐量**：单位时间内完成的进程数量，短作业有利于提高吞吐量
- **周转时间**：进程从创建到结束的时间，越短越好
- **等待时间**：进程在就绪队列中等待的时间，越短越好
- **响应时间**：交互式系统中，用户提交请求到系统首次响应的时间，越短越好

#### 调度算法(单核CPU)
###### 分类
- **非抢占式**：进程一旦开始运行，就会持续到阻塞或结束，不会被时钟中断打断
- **抢占式**：进程运行固定时间片/有高优先级进程到达，当前进程被强制挂起，调度器选择新进程

###### 1.先来先服务(FCFS)
- **规则**：按进程到达就绪队列的顺序，先到先执行，一直运行到阻塞/结束(非抢占式)
- **优点**：实现简单，**对长作业有利**
- **缺点**：**短作业等待时间长**，不利于I/O密集型场景
###### 2.最短作业优先(SJF)
- **规则**：优先选择运行时间最短的进程执行(可抢占/非抢占)
- **优点**：系统**吞吐量最高**，平均周转时间最短
- **缺点**：
	- **长作业可能饥饿**
	- 无法预知进程运行时间
###### 3.高响应比优先(HRRN)
- **公式**：`响应比 = (等待时间 + 要求服务时间) / 要求服务时间`
- **规则**：每次调度选择响应比最高的进程(非抢占式)
- **优点**：
	- **兼顾长短作业**
	- 长作业等待时间越久，响应比越高
	- 短作业要求服务时间越短，响应比越高
- **缺点**：每次调度需要计算响应比，开销较大
###### 4.时间片轮转(RR)
- **规则**：每个进程分配固定时间片(20ms~50ms)，时间片用完 / 阻塞 / 结束 就进行调度(抢占式)
- **优点**：公平性好，响应时间短，适合交互式系统
- **缺点**：
	- 时间片太短->频繁上下文切换，降低CPU效率
	- 时间片太长->退化为FCFS
###### 5.最高优先级优先(HPF)
- **规则**：选择优先级最高的进程执行(也分抢占式/非抢占式)
	- **静态优先级**：创建时确定，运行时不变
	- **动态优先级**：根据运行时间/等待时间调整
- **优点**：可灵活分配CPU资源，满足重要进程的需求
- **缺点**：
	- **低优先级进程可能饥饿**
	- 抢占式实现复杂

###### 6.多级反馈队列(MFQ)
- **规则**：
	- 设置多个队列，优先级从高到低，**优先级越高时间片越短**
	- 新进程先进入最高优先级队列，按**时间片轮转**执行
	- 若时间片用完仍未结束，则**降级**到下一级队列(**增长时间片**)
	- 若有新的进程加入高优先级队列，则**抢占式**调度 高优先级队列中的进程
- **优点**：综合时间片轮转+优先级调度，兼顾短作业响应快、长作业能完成，避免饥饿
- **缺点**：实现复杂，队列数量/时间片长度需合理配置



# 进程间通信(IPC)
#### 背景
- 每个进程各自拥有独立的虚拟地址空间，默认无法直接访问其他进程内存
- 通过内核作为中介实现数据交换和同步
- 所有IPC机制都依赖内核空间作为桥梁

#### 管道(Pipe)
###### 定义
- 管道是最基础的IPC方式，本质是**内核中的环形缓冲区**，数据遵循**先进先出**，**单向流动**(不支持lseek)，数据是无格式的流且大小受限(**缓冲区满则写阻塞，空则读阻塞**)。
###### 匿名管道(pipe `|` )
- 创建：`int pipe(int fd[2])`，返回读端`fd[0]`和写端`fd[1]`，仅存在内存，随进程结束销毁
- 通信范围：只能在有**父子关系的进程**间通信(`fork()`)
- 特点：
	- **单向通信**，若要双向需创建两个管道
	- **写端阻塞**直到数据被读走(**同步通信方式**)
- 典型场景：Shell命令 `ps aux | grep mysql`，管道两端的命令由shell创建出来的两个子进程分别执行
![[管道.PNG|258]]
###### 命名管道(FIFO)
- 创建：`mkfifo myPipe`，以文件形式存在于文件系统(类型为`p`)
- 通信范围：可在**无亲缘关系的进程**间通信
- 特点：
	- 单向通信，若要双向需创建两个管道
	- 写端阻塞直到数据被读走
###### 优缺点
- 优点：简单易用，内核自动同步读写
- 缺点：
	- **缓冲区满则写阻塞，空则读阻塞**，不适合频繁/大数据通信
	- 匿名管道只能用于亲缘进程间通信


#### 消息队列(Message Queue)

###### 定义
- 消息队列是内核维护的**消息链表**，以独立数据单元(消息体/数据块)传输
###### 核心特点
- **消息**是用户**自定义数据类型**，需约定格式，但是有大小限制
- **生命周期随内核**，进程结束后消息队列依然存在，需手动删除
- 支持频繁通信，可多次收发，像"进程间发邮件"
###### 优缺点
- 优点：支持**异步通信**，可缓存消息，适合频繁交互
- 缺点：
	- 有消息大小(MSGMAX) / 队列总长度(MSGMNB) 限制
	- 存在用户态->内核态->用户态的两次**数据拷贝**花销


#### 共享内存(Shared Memory)
###### 定义
- 将一块**物理内存**同时**映射**到多个进程的虚拟地址空间，实现**零拷贝**共享
- 现代常用`mmap()`映射文件进行实现
###### 核心特点
- 无需内核中转，**直接访问相同物理内存**，是**最快的IPC方式**
- 多个进程可同时读写，会引发**数据竞争**，必须配合同步机制(信号量)
###### 优缺点
- 优点：速度极快，无数据拷贝开销，适合大数据量共享
- 缺点：不提供同步机制，需手动处理并发冲突
![[共享内存.PNG|404]]


#### 信号量(Semaphore)
###### 定义
- 信号量本质是**整型计数器**，用于实现进程间的**互斥与同步**，不直接传输数据
###### 核心操作(原子操作)
- **P操作(wait)**：信号量-1，若结果<0，进程阻塞等待资源
- **V操作(signal)**：信号量+1，若结果<=0，唤醒一个阻塞进程
###### 典型用法
- **互斥信号量**：初始值为1，保证共享资源同一时间只被一个进程访问(解决共享内存冲突)
- **同步信号量**：初始值为0，保证进程执行顺序(如生产者与消费者)


#### 信号(Signal)
###### 定义
- 信号是进程间**异步通信机制**，用于通知进程发生了某件事情(异常/中断)
###### 核心特点
- 特点：
	- 异步性：可在任何时候送达
	- 简单性：只传递信号编号，不携带数据(实时信号除外)
	- 默认动作：具有默认操作
	- 不可靠性：传统信号可能丢失，实时信号(SIGRTMIN~SIGRTMAX)支持排队
- 来源：
	- **硬件**(`Ctrl+C`发送`SIGINT`)
	- **软件**(`kill`命令)
- 处理方式：
	- **执行默认操作**：如SIGTERM默认终止进程
	- **捕捉信号**：执行自定义信号处理函数
	- **忽略信号**：不做处理(特殊：`SIGKILL` / `SIGSTOP`无法捕捉和忽略)
###### 典型用法
- 终止进程：`kill - 9 PID`发送`SIGKILL`强制停止
- 暂停(挂起)进程：`Ctrl+Z`发送`SIGTSTP`
- 查看所有信号类型：`kill -l`
###### 优缺点
- 优点：异步、轻量、适合异常通知
- 缺点：不适合传输数据，仅用于事件通知

#### socket
###### 定义
- **可跨主机 / 网络**的IPC方式，也可用于本机进程间通信
###### 核心分类
1. **基于TCP(SOCK_STREAM)**：面向两家，可靠传输，适合大数据量、稳定通信
	- 流程：
		- 服务器：socket()--bind()--listen()--accept()--read()/write()--close()
		- 客户端：socket()--bind()--connect--read()/write()--close()
	 - ![[Socket(TCP).PNG|518]]
2. **基于UDP(SOCK_DGRAM)**：无连接，不可靠，适合高效、短消息实时传输
	- 流程：socket()--bind()--sendto()/recvfrom()--close()
	- ![[Socket(UDP).PNG|246]]
3. **本地Socket(AF_UNIX / AF_LOCAL)**：用于本机进程间通信，性能高于网络Socket
###### 优缺点
- 优点：可跨主机/网络，功能强大，支持多种通信模式
- 缺点：编程复杂，有网络开销(本地Socket除外)

#### 不同场景如何选择IPC？

|IPC 方式|通信范围|数据方向|效率|复杂度|主要用途|
|---|---|---|---|---|---|
|匿名管道|亲缘进程|单向|低|极低|简单父子进程通信|
|命名管道|任意进程|单向|低|低|无亲缘进程简单通信|
|消息队列|任意进程|双向|中|中|频繁异步消息通信|
|共享内存|任意进程|双向|最高|中高|大数据量共享（需同步）|
|mmap|任意进程|双向|最高|中|现代共享内存 / 文件高速 IO|
|信号量|任意进程|-|中|中高|进程互斥 / 同步（不传输数据）|
|信号|任意进程|单向|轻量|低|异常事件通知|
|Socket|本机 / 跨主机|双向|中 / 高|高|网络通信 / 本机高性能通信|

- 回答时要结合**效率(性能)**、**亲缘关系**、**数据量**、**是否跨机**、**复杂度**综合权衡



# 多线程同步

#### 竞争与协作
###### 前置概念
1. 并发的本质：
	- 单核CPU通过时间片轮转调度，让多个线程逐个执行，由于时间片很短，造成并发假象
	- 多线程共享进程资源(代码段、数据段、堆、文件等)，但每个线程有独立的栈和寄存器
2. **竟态条件**：
	- 多个线程同时操作**共享资源**，执行过程中发生**上下文切换**，导致操作被打断，最终结果不确定
3. **临界区**：
	- 定义：**访问共享资源的代码片段**
	- 要求：不能多个线程同时访问临界区，否则会引发数据混乱
###### 互斥
- 定义：保证**同一时间只有一个线程**进入临界区执行，避免竟态条件
- 作用：解决资源竞争问题，让临界区原子化(不可分割)，适用于多线程/多进程
###### 同步
- 定义：多线程/多进程需要**按约定顺序协作完成任务**，在关键点上互相等待/通知


#### 互斥与同步的实现与使用
#### 锁
###### 基本原理
- 线程进入临界区必须**加锁**，避免竟态条件
- 线程离开临界区必须**解锁**，释放资源
- 底层依赖CPU提供的**原子操作**，保证加锁 / 解锁操作不可分割
###### 忙等待锁(自旋锁)
- 实现：`while(TestAndSet(&lock->flag, 1) == 1)`循环自旋，直到获取到锁
- 特点：获取不到锁时**不主动放弃CPU**，一直空转消耗资源，直到调度程序调度
- 适用场景：锁持有时间短，多核环境，避免线程切换开销
###### 无等待锁(阻塞锁)
- 实现：获取不到锁时，将线程放入锁的**等待队列**，主动调用调度程序**让出CPU**
- 解锁时：从等待队列唤醒一个线程，将其加入就绪队列
- 特点：不浪费CPU资源，但是有线程切换开销
- 适用场景：锁持有时间长，单核环境

#### 信号量
###### 基础概念
- 本质：整型变量`sem`，代表**可用资源数量**
- 由操作系统提供的两个**原子操作**：**P操作(申请资源)**、**V操作(释放资源)**，保证操作不可中断
###### P/V操作规则
- P操作：`sem-=1`，若`sem<0`，线程阻塞，进入等待队列
- V操作：`sem+=1`，若`sem<=0`，唤醒等待队列中的一个线程
###### 信号量实现互斥
- 初始化信号量`mutex=1`，代表临界区空闲
- 线程进入临界区前执行`P(mutex)`，离开临界区时执行`V(mutex)`
- 效果：保证同一时刻只有一个线程进入临界区，实现互斥
###### 信号量实现同步
- 初始化信号量`s=0`，代表事件未发生
- 等待方执行`P(s)`，阻塞等待事件发生
- 通知方执行`V(s)`，唤醒等待的线程

#### 条件变量
###### 核心三要素
1. **互斥锁 mutex**：保护条件本身(条件也是共享变量)
2. **等待 wait**：**释放锁**->**休眠**->被唤醒后再**重新持有锁**
3. **唤醒 notify**：条件满足时唤醒等待线程
###### 固定流程
```
// 等待方 
lock(); 
while (条件不成立) {  
	cond.wait(lock); // 自动释放锁并阻塞 
} 
// 条件成立，重新持有锁，跳出while循环
do_something()

unlock();        // 做完任务后，解锁
```

```
// 唤醒方 
lock();            // 保护条件本身
修改条件; 
cond.notify_one(); // or notify_all()    // 唤醒阻塞线程
unlock();          // 解除对条件的保护 
```
###### 为什么必须用 while 而不是 if？
- 存在**惊群效应**：多个线程被同时唤醒
- 存在**虚假唤醒**：操作系统可能无故唤醒
- 必须**循环再检查一次条件**，确保安全

###### 特点
- 必须配合互斥锁一起使用
- 不提供数据传输，只做 “等待 + 唤醒”
- 适合：队列非空、缓冲区满、任务到达等场景


#### 经典同步问题
##### 生产者-消费者问题
###### 问题描述
- **生产者**：生成数据并放入固定大小的缓冲区；
- **消费者**：从缓冲区取出数据并处理；
- **约束**：
    1. 缓冲区满时，生产者必须等待；
    2. 缓冲区空时，消费者必须等待；
    3. 同一时刻只能有一个线程操作缓冲区（互斥）。

###### 同步机制（信号量）

|信号量|初始值|作用|
|---|---|---|
|`mutex`|1|互斥访问缓冲区（临界区）|
|`emptyBuffers`|N|记录空缓冲区数量，生产者用它判断是否可放数据|
|`fullBuffers`|0|记录满缓冲区数量，消费者用它判断是否可取数据|

- **互斥**：`mutex` 保证缓冲区操作原子性；
- **同步**：`emptyBuffers`/`fullBuffers` 实现「生产者等空位、消费者等数据」的顺序协作。

###### 核心代码实现
```
#define N 100
semaphore mutex = 1;          // 互斥信号量，用于临界区的互斥访问
semaphore emptyBuffers = N;   // 表示缓冲区「空槽」的个数
semaphore fullBuffers = 0;    // 表示缓冲区「满槽」的个数

// 生产者线程函数
void producer()
{
    while(TRUE)
    {
        P(emptyBuffers);        // 将空槽的个数 - 1（申请空位）
        P(mutex);               // 进入临界区（互斥访问缓冲区）
        将生成的数据放到缓冲区中;
        V(mutex);               // 离开临界区（释放互斥锁）
        V(fullBuffers);         // 将满槽的个数 + 1（通知消费者有数据）
    }
}

// 消费者线程函数
void consumer()
{
    while(TRUE)
    {
        P(fullBuffers);         // 将满槽的个数 - 1（申请满数据）
        P(mutex);               // 进入临界区（互斥访问缓冲区）
        从缓冲区里读取数据;
        V(mutex);               // 离开临界区（释放互斥锁）
        V(emptyBuffers);        // 将空槽的个数 + 1（通知生产者有空位）
    }
}
```

##### 哲学家就餐问题
###### 问题描述
- 5 个哲学家围坐圆桌，每两人之间放 1 支叉子（共 5 支）；
- 哲学家只有拿到**左右两边各 1 支叉子**才能吃面；
- 吃完后放下叉子继续思考；
- **风险**：若所有人同时拿起左边叉子，会发生**死锁**。
###### 方案一（朴素版，会死锁）
- 每个叉子对应一个信号量 `fork[i] = 1`；
- 哲学家先拿左叉 `P(fork[i])`，再拿右叉 `P(fork[(i+1)%N])`；
- 问题：所有哲学家同时拿左叉 → 全部阻塞在拿右叉 → 死锁。

###### 方案二（互斥锁版，低效）
- 加互斥信号量 `mutex = 1`，拿叉子前先 `P(mutex)`；
- 保证**一次只有一个哲学家**拿叉子，避免死锁；
- 问题：并发度极低，最多 1 人吃面，浪费资源。

###### 方案三（奇偶编号策略，推荐）
- 偶数编号哲学家：先拿左叉，再拿右叉；
- 奇数编号哲学家：先拿右叉，再拿左叉；
- 效果：打破循环等待，不会死锁，且最多 2 人同时吃面。

###### 方案四（状态检测版）
- 用数组 `state[]` 记录哲学家状态（THINKING/HUNGRY/EATING）；
- 只有**左右邻居都不在吃面**时，哲学家才能拿叉子吃面；
- 用 `test()` 函数检查并唤醒可吃面的哲学家。

###### 核心代码实现
```
// 方案三：奇偶编号

#define N 5
semaphore fork[5];   // 每个叉子一个信号量，初值为 1

// i 为哲学家编号 0-4
void smart_person(int i)
{
    while(TRUE)
    {
        think();                // 哲学家思考
        if ( i % 2 == 0 )       // 偶数编号：先拿左叉，后拿右叉
        {
            P(fork[i]);             // 去拿左边的叉子
            P(fork[(i + 1) % N ]);  // 去拿右边的叉子
        }
        else                    // 奇数编号：先拿右叉，后拿左叉
        {
            P(fork[(i + 1) % N ]);  // 去拿右边的叉子
            P(fork[i]);             // 去拿左边的叉子
        }
        eat();                 // 哲学家进餐
        V(fork[i]);             // 放下左边的叉子
        V(fork[(i + 1) % N ]);  // 放下右边的叉子
    }
}
```


```
// 方案四：状态检测

#define N 5
#define LEFT (i + N - 1) % N   // i 的左边邻居编号
#define RIGHT (i + 1) % N      // i 的右边邻居编号
#define THINKING 0             // 思考状态
#define HUNGRY 1               // 饥饿状态
#define EATING 2               // 进餐状态

int state[N];                 // 数组记录每个哲学家的状态
semaphore s[5];               // 每个哲学家一个信号量，初值 0
semaphore mutex;             // 互斥信号量，初值为 1

// i 为哲学家编号 0-4
void test(int i)
{
    // 如果 i 号的左边右边哲学家都不是进餐状态，把 i 号哲学家标记为进餐状态
    if ( state[i] == HUNGRY &&
         state[LEFT] != EATING &&
         state[RIGHT] != EATING )
    {
        state[i] = EATING;     // 两把叉子到手，进餐状态
        V(s[i]);               // 通知第 i 哲学家可以进餐了
    }
}

// 功能：要么拿到两把叉子，要么被阻塞起来
void take_forks(int i)         // i 为哲学家编号 0-4
{
    P(mutex);               // 进入临界区
    state[i] = HUNGRY;      // 标记哲学家处于饥饿状态
    test(i);                // 尝试获取 2 支叉子
    V(mutex);               // 离开临界区
    P(s[i]);                // 没有叉子则阻塞，有叉子则继续正常执行
}

// 功能：把两把叉子放回原处，并在需要的时候，去唤醒左邻右舍
void put_forks(int i)         // i 为哲学家编号 0-4
{
    P(mutex);               // 进入临界区
```

##### 读者-写者问题
###### 问题描述
- **读者**：只读取数据，可并发读；
- **写者**：可读写数据，必须独占访问；
- **约束**：
    1. 读 - 读：允许并发；
    2. 读 - 写：互斥，不能同时进行；
    3. 写 - 写：互斥，不能同时进行。
###### 策略 1：读者优先
- 只要有读者在读，后续读者可直接进入，写者必须等待；
- 风险：写者可能**饥饿**（一直没机会写）。
###### 策略 2：写者优先
- 只要有写者在写 / 等待，后续读者必须等待，写者优先执行；
- 风险：读者可能饥饿。
###### 策略 3：公平策略
- 用额外信号量 `flag` 限制读者特权，保证读写公平，避免饥饿。
- 流程：
1. **多个读者先到**
    - 依次 `P(flag)` → 进门 → **立刻 `V(flag)`**
    - 第一个读者 `P(wDataMutex)` 锁住写者
    - 后面读者直接并发读
    - → **读读并发，没问题**
2. **此时写者到达**
    - 写者 `P(flag)` 成功拿到 flag（因为读者都释放了）
    - 但尝试 `P(wDataMutex)` 时**阻塞**（读者还在读）
    - → **写者排队等待，并且持有 flag**
3. **后续读者 / 写者到达**
    - 全部因为 `flag` 被写者持有 **阻塞在门口**
    - → **不会插队，保证公平排队**
4. **最后一个读者退出**
    - `V(wDataMutex)` 释放
    - 写者被唤醒，开始写
5. **写者写完**
    - **释放 / 唤醒**：`V(wDataMutex)` + `V(flag)`
    - 后面排队的读者 / 写者才能继续

###### 核心代码实现

```
// 读者优先

semaphore wMutex;         // 控制写操作的互斥信号量，初始值为 1
semaphore rCountMutex;    // 控制对 Rcount 的互斥修改，初始值为 1
int rCount = 0;           // 正在进行读操作的读者个数，初始化为 0

// 写者进程/线程执行的函数
void writer()
{
    while(TRUE)
    {
        P(wMutex);         // 进入临界区（写互斥）
        write();           // 写数据
        V(wMutex);         // 离开临界区（释放写互斥）
    }
}

// 读者进程/线程执行的函数
void reader()
{
    while(TRUE)
    {
        P(rCountMutex);    // 进入临界区（保护读者计数）
        if ( rCount == 0 )
        {
            P(wMutex);     // 如果有写者，则阻塞写者（第一个读者锁写者）
        }
        rCount++;           // 读者计数 + 1
        V(rCountMutex);    // 离开临界区（释放读者计数锁）

        read();             // 读数据

        P(rCountMutex);    // 进入临界区（保护读者计数）
        rCount--;           // 读完数据，准备离开
        if ( rCount == 0 )
        {
            V(wMutex);     // 最后一个读者离开了，则唤醒写者
        }
        V(rCountMutex);    // 离开临界区（释放读者计数锁）
    }
}
```

# 死锁
#### 定义
- 多个线程 / 进程**互相持有对方需要的资源**，并在不释放自身资源的情况下等待对方释放资源，导致**无限阻塞**、无法继续执行的现象
#### 死锁的四个必要条件(必须同时满足)
1. **互斥条件**：共享资源不能被多个线程同时使用
2. **持有并等待条件**：线程已经持有至少一个资源，又去申请其他资源，且不释放已持有的资源
3. **不可剥夺条件**：线程已持有的资源，在使用完之前不能被其他线程剥夺，只能持有者主动释放
4. **环路等待条件**：线程获取资源的线路构成**环形等待链**
#### 死锁的排查(?)
- **`pstack <pid>`**：查看线程调用栈，发现线程长期阻塞在 `pthread_mutex_lock`
- **`gdb -p <pid>`**：
    - `info thread`：查看所有线程状态
    - `thread <id>`：切换到阻塞线程
    - `bt`：打印函数调用栈
    - `p mutex`：查看锁的持有者，确认互相等待关系
#### 避免死锁的方法
核心：只要破坏四个必要条件中的**任意一个**，就能避免死锁，最常用、可行的是**破坏环路等待条件**
1. **资源有序分配法**：规定所有线程必须按照**相同的顺序申请资源**
2. 破坏 “持有并等待”：一次性申请所有需要的资源，要么全拿到，要么一个都不拿
3. 破坏 “互斥”：某些资源可改为共享（比如只读资源），但大部分场景不适用
4. 破坏 “不可剥夺”：允许操作系统强行剥夺线程持有的资源（代价高，少用）


# 悲观锁与乐观锁
#### 锁的整体体系
- 背景：多线程访问共享数据时，为避免数据错乱，需要在访问前加锁
- 分类：常见的锁类型：**互斥锁、自旋锁、读写锁、悲观锁、乐观锁**
- 选择：不同的锁适用于不同的业务场景，选择时考虑：
	- **冲突概率**(加锁的性能开销)
	- **锁的持有时间**
	- **读写比例**
- 原则：无论哪种锁，都要**尽量减小加锁粒度**（只锁必要代码段），提升执行速度
#### 互斥锁 vs 自旋锁
- 核心区别(**加锁失败后的处理方式**)：
	- 互斥锁：加锁失败后，线程**主动释放CPU**，进入**睡眠状态**，锁释放后由内核唤醒线程，存在**两次线程上下文切换**的开销(挂起+恢复)
	- 自旋锁：加锁失败后，线程**忙等待**(循环重试)，**不主动释放CPU**，**无上下文切换开销**
- **实现与原理**：
	- 互斥锁：操作系统**内核实现**的独占锁
	- 自旋锁：基于CPU的**CAS原子指令**在**用户态**实现，将检查锁状态+设置所持有合并为一条不可分割的指令，保证原子性
- **适用场景**：
	- 互斥锁：锁持有时间较长，单核CPU场景
	- 自旋锁：**锁持有时间极短**(临界区执行时间<上下文切换时间)，多核CPU场景(单核CPU下自旋锁一直占用CPU)

#### 读写锁
- 核心设计：
	- 分为读锁和写锁，而且是**基于 互斥锁/自旋锁 实现的**
	- **读锁**：**共享锁**，多个线程可同时持有读锁，读取共享资源
	- **写锁**：**独占锁**，同一时刻只能被一个线程持有，修改共享资源
	- 规则：读-读并发、读-写互斥、写-写互斥
- 优先级策略：
	- **读优先锁**：优先让读线程获取锁，读并发更高，但是写线程可能**饥饿**
	- **写优先锁**：优先让写线程获取锁，但是读线程可能**饥饿**
	- **公平读写锁**：使用队列让读写线程按照**先进先出**排队，避免饥饿
- 适用场景：明确区分**读多写少**的场景，能大幅提升读并发效率

#### 悲观锁 vs 乐观锁
- 悲观锁：
	- 心态：认为并发冲突概率高，必须**先加锁再操作**，防患于未然
	- 具体分类：互斥锁、自旋锁、读写锁
	- 流程：加锁->操作资源->解锁
- 乐观锁：
	- 心态：认为并发冲突概率低，先修改资源，再验证冲突，事后补救
	- 本质：**全程不加锁**(无锁编程)，依赖版本号或CAS验证
	- 实现方式：
		- **版本号机制**：资源带版本号，提交时对比版本号，一致则更新，不一致则放弃
		- **CAS指令**：原子比较内存值和旧值，一致则替换为新值，否则重试
	- 典型例子：在线文档多人协作、Git/SVN 版本控制等
