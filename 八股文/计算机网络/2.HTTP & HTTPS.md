# HTTP简述与特性
## 简述
**Web** 上的通信都是建⽴在 HTTP 协议上的(Web借助WWW实现)
1. 客户端发起 HTTP 请求
2. 服务器做出响应处理后，返回 HTTP 响应报⽂
[[4.IP#1.DNS 域名解析|DNS解析过程]]

**WWW 构建技术**（3 项）：
1. HTML (HyperText Markup Language)：作为⻚⾯的⽂本标记语⾔
2. HTTP (HyperText Transfer Protocol)：⽂档传递协议；
3. URL (Uniform Resource Locator)：指定⽂档所在地址

**HTTP 版本**：
1. 1996 年的 5 ⽉，HTTP/1.0，并记载于 RFC1945。现在 HTTP/1.0，仍然被使⽤在服务器端；
2. 1997 年 1 ⽉，发布 HTTP/1.1 ，是⽬前主流的 HTTP 协议版本
3. 现今，HTTP/2.0 正在制订中，但还未得到⼴泛的使⽤
## 特性
1. **简单**
- 基本报文格式：header+body，header：key-value简单文本形式，易于理解

2. **灵活+易于扩展**
- HTTP协议中的请求方法、URI/URL、状态码、头文段可自定义和扩充
- HTTP工作在应用层，下层可随意变化
- HTTPS=HTTP与TCP之间增加SSL/TSL安全传输层，HTTP/3=TCP换成基于UDP的QUIC

3. **无状态**
- 服务器不会记忆HTTP的状态=》减轻服务器负担/关联性操作比较麻烦
- 解决方法：Cookie=》在请求和相应报文中添加Cookie，管理客户端状态

4. **明文传输**
- 方便阅读和调试(浏览器F12/Wireshark抓包)
- 信息透明，容易被窃取

5. **不安全**
- 明文传输=》窃听
- 不验证身份=》伪装
- 无法验证报文的完整性=》篡改
- 解决办法：HTTPS，引用SSL/TSL层保障安全

## HTTP请求方法
1. **GET**
	- **用来获取数据**
	- 明文参数在 URL 里
	- **幂等**：执行多次结果一样（不会改服务器数据）
	- 可缓存
	- 有长度限制
 2. **POST**
	- **用来提交数据、创建资源**
	- 参数在请求体里，更安全
	- **不幂等**：多次提交可能创建多条数据
	- 不默认缓存
	- 数据大小无限制
 3. **PUT**
	- **更新资源（完整替换）**
	- 幂等：多次执行结果一样
	- 一般用于修改整个资源
 4. **DELETE**
	- **删除资源**
	- 幂等
 5. **HEAD**
	- 和 GET 一样，但**只返回响应头，不返回 body**
	- 用来检查资源是否存在、是否过期
6. **OPTIONS**
	- 预检请求（CORS 跨域时浏览器自动发）
	- 询问服务器支持哪些方法、跨域是否允许

## GET 和 POST 的区别
1. **GET 用来查，POST 用来提交 / 创建**
2. GET 参数在 URL，POST 在请求体
3. GET 可缓存，POST 一般不缓存
4. GET 幂等，POST 不幂等
5. GET 有长度限制，POST 没有
6. GET 历史记录会保存，POST 不会

## HTTP状态码
**1xx 信息类**（很少考）
- 100 Continue：继续发送

 **2xx 成功**
- **200 OK**：成功
- **201 Created**：创建成功（POST/PUT）
- **204 No Content**：成功，但没返回内容

 **3xx 重定向**
- **301 永久重定向**：网址搬家了
- **302 临时重定向**
- **304 Not Modified**：缓存未过期（协商缓存）

 **4xx 客户端错误**
- **400 Bad Request**：请求参数错误
- **401 Unauthorized**：未登录 / 没 token
- **403 Forbidden**：登录了但没权限
- **404 Not Found**：资源不存在
- **405 Method Not Allowed**：方法不支持（比如用 POST 访问只支持 GET 的接口）

**5xx 服务器错误**
- **500 Internal Server Error**：服务器代码崩了
- **502 Bad Gateway**：网关错误
- **503 Service Unavailable**：服务器过载 / 维护
- **504 Gateway Timeout**：网关超时


# HTTP版本演变
## HTTP/0.9
在1991年就已经发布，只⽀持 GET ⽅法，也没有请求头，服务器只能返回 HTML格式的内容。
## HTTP/1.0
1. 引⼊了请求头和响应头，⽀持多种请求⽅法和状态码
2. **不⽀持持久连接**，每次请求都需要建⽴新的连接
## HTTP/1.1
### HTTP/1.1的特性
1. **持久连接**
- 解决HTTP/1.0每次请求都需要建立新的连接的问题
- 只要客户端和服务器任意一端没有明确提出断开连接，则保持TCP连接状态

2. **管道网络传输**
- 同一个TCP连接中，客户端可以**连续**发起多个请求，不必等待服务器的响应
- 但是服务器必须按照请求的顺序发送响应！！！
- 管道网络传输**解决了请求的队头阻塞**，但是**没有解决响应的队头阻塞**
- 注：HTTP/1.1管道网络传输默认不开启，且浏览器基本不支持

3. **队头阻塞**
- 请求序列中的请求A因某种原因被阻塞而无法响应时，在后面排队的请求BCD也一同被阻塞(为了保证响应顺序)，导致客户端一直收不到数据
### HTTP/1.1的不足
1. 头部冗余：每个请求和相应都需要带有一定的头部信息(因与之前相同而冗余)
2. 队头阻塞(响应端)
3. 没有请求优先级控制
4. 请求只能从客户端开始，服务器只能被动响应

## HTTP/2
### 相较HTTP/1.1的改进
1. **头部压缩**：HTTP/2 使⽤ **HPACK** 压缩算法(静态表+动态表+Huffman编码)对请求和响应头部进⾏压缩，减少了传输的头部数据量，降低了延迟。

2. **⼆进制帧**：HTTP/2 将数据分割成⼆进制帧进⾏传输(HTTP/1.1采用文本格式)，分为头信息帧（Headers Frame）和数据帧（DataFrame），增加了数据传输的效率。

3. **并发传输**：
- 引出了 Stream 概念，**多个Stream** 复⽤在⼀条 TCP 连接，针对不同的 HTTP 请求⽤独⼀⽆⼆的**Stream ID** 来区分
- 接收端可以通过 Stream ID 有序组装成 HTTP 消息=》不同 Stream 的帧是**可以乱序发送**的(**解决响应的队头阻塞问题**)
- HTTP/2通过 Stream 实现的并发比HTTP/1.1通过 连接 实现的并发更牛逼：当HTTP/2实现100个并发Stream时，只需要建立一次TCP连接 <=> 而HTTP/1.1需要建立100个TCP连接(管道网络传输已弃用)，每个TCP连接都要经过TCP握手、慢启动以及TLS握手过程，这些都是很耗时的。

4. **服务器推送**：在 HTTP/2 中，服务器可以对客户端的**⼀个请求发送多个响应**，即服务器可以额外的向客户端推送资源，⽽⽆需客户端明确的请求。

### HTTP/2的不足
1. **TCP造成**的**队头阻塞**问题：
- HTTP/2基于TCP协议传输数据
- TCP协议是字节流协议，必须保证收到的字节完整且连续，客户端内核才会将缓冲区中的数据返回给HTTP应用
- 前一个字节数据未到达，导致持续的阻塞

2. **TCP和TLS的握手时延迟**
- TCP三次握手 + TLS1.2 = 1+1RTT

3. **网络迁移时需要重新连接**
- IP地址或端⼝发⽣变化时，需要重新进⾏连接

## HTTP/3
### 相较HTTP/2的改进 or QUIC协议的特点
1. **基于UDP协议** 在 应用层 实现了**QUIC协议**
- QUIC协议有类似TCP的连接管理、拥塞窗口、流量控制
- 相当于在应用层将不可靠的UDP协议变为可靠的了
2. **快速建立连接**
- HTTP/3 首次(TCP+TSL1.3)：1 RTT
- HTTP/3 重连：0 RTT
3. **无队头阻塞**
- QUIC使用UDP协议进行数据传输，一个连接上的多个stream之间没有依赖，互不影响
4. **连接迁移**
- 基于TCP的HTTP是通过四元组(源IP、源端口、目的IP、目的端口)确定连接的
- QUIC协议通过**连接ID**来标记通信的两个端口
- 只要仍保有上下文信息，就可以使用连接ID来无缝复用原连接
5. **向前纠错能力**
- 数据包=本身内容+部分其他数据包的数据
- 少量的丢包可以通过其他包的冗余数据进行组装，而无需重传
- 牺牲部分一次发送数据的上限 / 减少丢包导致的重传

# HTTP缓存
**定义**：对于已经请求过的资源，客户端/代理服务器将副本保存在本地，下次请求同一资源时，先检查缓存中是否存在有效的副本，如果有则直接读取，不必通过网络获取服务器的响应。

**分类**：强制缓存和协商缓存

## 强制缓存
**定义**：浏览器自行判断是否命中有效缓存，命中则直接读取目标资源，无需与服务器通讯
**分类**：Expires强缓存和Cache-Control强缓存

### Expires强缓存
**定义**：
- 服务器返回强缓存**到期时间点**
- 在该时间范围内，客户端通过获取本地时间戳进行时间比较，有效则可以直接从缓存中读取目标资源。
**漏洞**：
- 本地时间不准确，可能导致资源一直无效或一直有效
- 由于该原因，Expires强缓存现在基本弃用
### Cache-Control强缓存
**定义/改进**：服务器返回强缓存**有效时间长度**(相对时间)，从客户端接收到资源开始计时
**属性**：
- max-age：客户端缓存有效期N秒
- s-maxage：代理服务器缓存有效期N秒
- **no-cache**：强制进行协商缓存
- no-store：禁止任何缓存策略
- public：可被浏览器和代理服务器缓存
- private：只能被浏览器缓存(默认)

**流程**：
- 首次请求：服务器返回资源+Cache-Control(设置过期时间)，浏览器缓存资源
- 再次请求：浏览器计算缓存是否过期；未过期则直接读取缓存，过期则发起新请求

## 协商缓存
**定义**：通过服务器告知客户端是否可以使用缓存的方式
**分类**：基于Last-Modified和If-Modified-Since的协商缓存 < 基于Etag的协商缓存

### 基于Last-Modified和If-Modified-Since的协商缓存
**流程**：
- 服务器读出**文件修改时间**
- 将文件修改时间赋给响应头的**Last-Modified**字段
- 设置Cache-Control：**no-cache**
- 客户端读取到Last-Modified字段时，会在下次的请求头中携带字段：**If-Modified-Since**(=Last-Modified的时间)
- 之后每次都资源的请求都会带上If-Modified-Since字段，服务器通过比较If-Modified-Since和该资源的最后修改时间进行判断
- 如果最后修改时间较新：资源被修改过，返回最新资源，响应HTTP 200 OK
- 如果最近修改时间较旧：资源无新修改，响应HTTP 304，走缓存

**缺点**：
- 文件内容不修改的情况下，依旧可能更新文件修改时间(修改文件名)=》文件内容无修改，缓存却失效了
- 文件在极短时间内完成修改的时候，文件修改时间(单位：秒)不变，导致不会返回新的文件
- 解决办法：基于Etag的协商缓存
### 基于ETag的协商缓存
**流程**：
- 首次请求：服务器读取文件+计算**文件指纹**(根据文件内容计算出的唯一哈希值)，将文件指纹放在响应头的ETag字段，和资源一起返回给客户端
- 再次请求：
	- 客户端从缓存中读取上次返回的ETag(即文件指纹)，赋给请求头的If-None-Match字段，将上一次的文件指纹与本次请求一起发给服务器
	- 服务器拿到请求头中的If-None-Match字段(即上一次的文件指纹)，并再次读取资源并生成文件指纹，通过对比文件指纹进行判断
	- 文件指纹完全吻合：文件未修改，返回 HTTP 304，走缓存
	- 文件指纹不吻合：文件已修改，将新的文件指纹重新存储到响应头的ETag中，与资源一起返回给客户端

**缺点**：
- 计算文件指纹=》服务器的计算开销 ↑ =》如果文件尺寸大，数量多 =》影响服务器性能
- ETag分为强验证和弱验证(权衡)：
	- 强验证：生成的哈希码深入每个字节，可以保证文件内容绝对不变，但是非常消耗计算性能
	- 弱验证：生成的哈希码只涉及文件的部分属性，生成速度快，但文件准确率不高

**注意**：
- **优先级**： Etag > Last-Modified
- 协商缓存需要配合强制缓存的Cache-Control=**no-cache**字段来使用
- **只能在未命中强制缓存的时候，才能发起协商缓存的请求**

# HTTPS
**定义**：HTTPS = HTTP + 加密 + 认证 + 完整性校验  =》 防窃听 + 防冒充 + 防篡改

## 两种加密方法
### 对称加密
- **定义**：一把钥匙，加密和解密都用它
- **优点**：速度快，适合大量数据传输
- **缺点**：钥匙怎么安全地交给对方？
### 非对称加密
- **定义**：
	- 两把钥匙：公钥(可公开给所有人) + 私钥(自己持有 绝不泄露)
	- 公钥加密，私钥解密：保证数据无法被窃取
	- 私钥加密，公钥解密：保证传输方的身份
- **优点**：公钥可以随便发，解决钥匙怎么传的问题
- **缺点**：计算慢，不适合加密大文件

## 建立连接的流程(混合加密)
- 三次握手建立连接
- 服务器生成公钥+私钥
- 服务器将公钥+自己的信息(域名、公司名)提交给CA
- CA使用自己的私钥，给**服务器的公钥**做"数字签名"，生成**数字证书**
- 服务器将数字证书发给客户端
- 客户端的浏览器/系统中**预先存有CA的公钥**，使用它解开证书，验证签名
	- 能解开=》证书是真的，服务器可信
	- 解不开=》证书是假的，服务器不可信
- 客户端解开数字证书后，确认服务器可信，并**拿到服务器公钥**
- 客户端**生成会话密钥**，并**使用服务器公钥加密**，再发给服务器
- 服务器**使用自己的私钥解开**，拿到会话密钥
- 之后**使用会话密钥**进行数据的加密传输

## 三个保障方式
1. **防窃听**
- 借助CA验证身份，并传递会话密钥，进行对称加密传输
2. **防篡改**
- **摘要算法**：给数据生成一个唯一的指纹(MD5、SHA-256)
3. **防冒充**
- 服务器用**私钥**给数据的指纹加密(**数字签名**)，客户端用服务器的公钥解开后，自己再算一遍指纹
	- 两个指纹一样=》数据没被改，来源是真服务器
	- 数据被篡改了=》直接丢弃
