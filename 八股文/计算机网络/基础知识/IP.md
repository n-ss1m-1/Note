# IP的基本认识
##### IP的定位与作用
- **层级**：IP位于TCP/IP模型的第三层(**网络层**)
- **核心作用**：实现**主机与主机**之间**端到端**(end to end)通信
##### IP与MAC的关系
- MAC(数据链路层)：负责**同一网段内相邻两个节点**的通信，相当于`区间车票`，只在一段链路内生效，**源/目标MAC地址会随每一跳路由器而变化**
- IP(网络层)：负责跨网段的远程定位，相当于`完整行程表`，在无NAT的网络中，**源/目标IP在整个传输过程中保持不变**
- **协作关系**：二者缺一不可

# IP地址的基础知识
###### 定义
- 本质：IPv4地址是**32位二进制数**，为方便记忆采用**点分十进制**表示(每8位为一组，转为十进制后使用`.`分隔)
- 总量：理论最大地址数：pow(2,32) = 4294967296(约 43 亿)，实际可用数更少
- 分配对象：以**网卡**为单位进行分配，一台设备可拥有多个IP地址
- 地址耗尽缓解方法：**NAT技术**，让多台设备共享公网IP，突破地址限制数量
#### 分类(A/B/C/D/E类)

| 类别  | 地址范围                        | 结构                   | 最大主机数                | 用途    |
| --- | --------------------------- | -------------------- | -------------------- | ----- |
| A 类 | 0.0.0.0 ~ 127.255.255.255   | 网络号（7 位）+ 主机号（24 位）  | pow(2,24)−2=16777214 | 大型网络  |
| B 类 | 128.0.0.0 ~ 191.255.255.255 | 网络号（14 位）+ 主机号（16 位） | pow(2,16)−2=65534    | 中型网络  |
| C 类 | 192.0.0.0 ~ 223.255.255.255 | 网络号（21 位）+ 主机号（8 位）  | pow(2,8)−2=254       | 小型网络  |
| D 类 | 224.0.0.0 ~ 239.255.255.255 | 组播地址（28 位）           | 无                    | IP 多播 |
| E 类 | 240.0.0.0 ~ 255.255.255.255 | 保留（28 位）             | 无                    | 预留备用  |
![[IP地址分类.PNG|696]]
###### 特殊地址
- **主机号全为1**：**广播地址**(发给网络/网段内所有主机)
- **主机号全为0**：**网络地址**(代表整个网段)
###### 广播
- **本地广播**：在本网络内广播，路由器不转发到其他网络
- **直接广播**：目标地址为其他网络(不同网络之间)，可跨网络传播(存在安全风险，默认不转发)
###### 多播/组播(D类)
- 用于向特定组内的主机发送数据包，可穿透路由器
- 224.0.0.0 ~ 224.0.0.255：局域网预留组播
- 224.0.1.0 ~ 238.255.255.255：互联网可用组播
- 239.0.0.0 ~ 239.255.255.255：本地管理组播
###### 区分单播-广播-组播
- **单播**：精准的 “一对一私聊”，只发给指定那一个人。(**TCP/UDP**)
- **广播**：粗暴的 “一网打尽”，适合局域网内发现服务，但跨网会被隔离。(**仅UDP**)
- **组播**：精准的 “分组推送”，是高效实现一对多通信的核心技术。(**仅UDP**)
![[单播-广播-组播.PNG|384]]

######  IP分类的优点和缺点
- 优点：
	- 结构简单，路由器选路效率高(0 / 1判断)
- 缺点：
	- 同一网络下没有地址层次(ABCDE之后不能再分)，缺少灵活性
	- 地址浪费严重：B类IP数冗余，C类IP数不足

#### 无分类域间路由 CIDR
###### 背景
- 问题：传统分类 IP (A/B/C 类)地址粒度固定，灵活性差，易造成地址浪费(如 B 类地址主机数过多、C 类过少)。
- 解决方法：提出 无分类域间路由(CIDR)，废除固定分类，灵活划分网络号和主机号
###### CIDR表示方法
- 表示形式：`a.b.c.d/x`
	- `a.b.c.d`：标注IPv4地址
	- `/x`：前缀长度，代表**前x位为网络号**，剩余32-x位为主机号
- 等价表示：**子网掩码**（如`/24`等价于`255.255.255.0`）
	- 计算方式：将IP地址于子网掩码做**按位与(&)运算**，即可得到网络号
###### 子网划分
- 核心思想：从主机号中**借位**作为子网号，将一个大网段拆分成为多个更小的子网
- 示例：对 `192.168.1.0/24` 借 2 位主机位作为子网号（即 `/26`）：
- 子网数：pow(2,2)=4 个
- 每个子网可用主机数：26−2=62 个(除去主机号全0和全1)
![[子网划分.PNG|366]]
###### 优势
- **地址利用率高**：可按需分配网段大小，避免地址浪费。
- **路由聚合能力强**：将多个连续网段聚合为一条路由表项，减小路由表规模，提升转发效率。

##### IP路由控制
- 路由控制表：维护映射关系，目标网络地址 -> 下一跳路由器/接口地址
- **最长匹配原则**：当路由表中存在多条匹配目标地址的记录时，选择**网络号前缀最长**的那条记录，保证转发的精准性
- **环回地址**：
	- 地址范围：**127.0.0.1**(典型)，整个127.0.0.0/8网段为环回地址
	- 作用：用于**同一台计算机内部进程间通信**，数据包不会流出本机
	- 等价主机名：localhost
##### IP分片与重组
- **MTU（最大传输单元）**：IP 分片的根本依据，每种数据链路层协议规定的帧最大承载容量不同（(如以太网 MTU = 1500 字节)）
- **分片触发条件**：当 **IP 数据包大小 > MTU** 时，发送方必须将大数据包拆分为多个小数据包(分片)，才能在链路上传输，然后由接收方重组。
- **分片丢失的后果**：分片传输中若**任意一个分片丢失**，接收方无法重组，将导致**整个 IP 数据包(及所有分片)失效**，只能直接丢弃/上层协议重传。
- **传输策略建议**：
	- **TCP 协议**：引入 MSS(最大分段大小)，在 TCP 层进行分片，只重传丢失的特定分片。
	- **UDP 协议**：由于 UDP 不可靠，**尽量不要发送大于 MTU 的数据包**，防止分片丢失导致数据完全丢失(UDP不重传)。
#### IPv6
###### 背景
- 地址枯竭
###### 地址结构与表示方法
- 长度：128位，以16位为一组(十六进制表示)，用冒号`:`分隔
- 压缩规则：连续的0可以省略，并用两个冒号`::`表示，但一个地址中只能出现一次
###### 地址分类

| 地址类型     | 表示形式        | 用途             | 类比 IPv4             |
| :------- | :---------- | :------------- | :------------------ |
| **单播地址** | -           | 网络节点独立标识       | 公有 / 私有 IP          |
| - 链路本地单播 | `FE80::/10` | 同一链路内通信，不经过路由器 | 无（IPv4 无类似）         |
| - 唯一本地地址 | `FC00::/7`  | 内网单播通信         | 私有 IP (192.168.x.x) |
| - 全局单播地址 | -           | 互联网通信          | 公有 IP               |
| **组播地址** | `FF00::/8`  | 一对多通信          | 广播组播                |
| **任播地址** | -           | 通信一组地址中路由最近的节点 | 无                   |
| **环回地址** | `::1/128`   | 本机通信           | `127.0.0.1`         |
|          | `::/128`    | 未定义            | 无                   |

**注意**：IPv6 **没有广播地址**，用组播替代了广播功能，减少网络开销。

###### IPv6的优势
- 自动配置：即插即用，无需DHCP服务器也能自动获取IP
- 安全性：原生支持应对伪造IP地址的网络安全功能以及防监听功能，安全性更高
- 取消首部校验和字段：数据链路层和传输层都会校验
- 取消分片/重组相关字段：只允许在源/目标主机进行，不允许在路由器上进行，提升了路由器转发速率
- 取消选项字段：首部长度固定为40字节，选项可能出现在IPv6首部的`下一个首部`指出的位置上

# IP协议相关技术

#### 1.DNS 域名解析
###### DNS是什么
**DNS**(Domain Name System)是一种将域名转换为IP地址的**分布式系统**。(域名易于记忆，IP不易记忆)
###### 分布式设计的原因(集中式设计的问题)
1. 单点故障 -> 网络瘫痪
2. 通信容量巨大 -> 一台服务器难以满足
3. 远距离查询 -> 严重时延
4. 维护成本大
###### 域名的层级关系
分隔符：句点( . )
级别：越靠右的位置，层级越高(越靠近根)
###### DNS解析过程(查到即返回)
1. 查询**浏览器缓存**
2. 查询**本地计算机缓存**(Host文件)
3. 主机向**本地DNS服务器**发送DNS查询请求
4. 本地DNS服务器查看自身是否**缓存**该域名的IP地址
5. 本地DNS服务器向**根DNS服务器**发送查询请求，根DNS服务器不负责解析域名，但是会告诉本地DNS服务器应该前往哪个顶级域名DNS服务器查询下一步的信息
6. 本地DNS服务器接着向指定的**顶级域名DNS服务器**发送查询请求，该顶级域名DNS服务器不负责解析域名，但是会告诉本地DNS服务器应该前往哪个权威DNS服务器查询下一步的信息
7. 本地DNS服务器最后向指定的**权威DNS服务器**发送查询请求，权威DNS服务器返回查询结果给本地DNS服务器
8. 本地DNS服务器将收到的IP地址返回给浏览器，并将域名解析结果缓存在本地
9. 浏览器使用IP地址发起连接，获取网页内容

![[DNS解析过程.png|450]]

###### 递归查询与迭代查询
1. **递归查询**
	- DNS客户端只需发送一个查询请求，然后等待完整的解析结果。
	- 上层DNS服务器会自行查询下一级的服务器，并将最终结果返回给客户端。
2. **迭代查询**
	- DNS客户端只询问上层DNS服务器A一个更底层的DNS服务器B的地址，然后再自行向DNS服务器B发起查询请求，以此类推，直到获取完整的解析结果为止。
3. 递归查询用于客户端(用户)，迭代查询用于DNS服务器之间的通信。


#### 2.ARP 与 RARP 协议
###### ARP的定位
- 位于**网络层和数据链路层**之间，解决**已知下一跳IP地址，如何获取其MAC地址**的问题
###### ARP的工作流程
1. **ARP请求(广播)**：主机以广播形式发送ARP请求，本网段内所有设备都会收到该请求
2. **ARP响应(单播)**：只有IP地址匹配的设备才会响应，单播返回自己的MAC地址
3. **ARP缓存**：主机获取MAC地址后会存入**ARP缓存**，下次通信直接读取，避免重复广播；缓存有效期过期后，自动清理
![[ARP协议.png|257]]
###### RARP协议(ARP逆过程)
- 核心作用：**已知MAC地址，请求对应的IP地址**
- 工作流程：
	1. 设备广播RARP请求，获取IP地址
	2. RARP服务器查询本地映射表，返回该MAC地址对应的IP地址
	3. 设备收到响应，配置自身IP地址
- 应用场景：无盘工作站、打印机等嵌入式设备，自身无法存储IP地址
- 替代：如今RARP已逐步被**DHCP**等协议替代


#### 3.DHCP 动态获取 IP 地址
###### 作用
- **动态分配IP地址**：自动为客户端分配IP地址、子网掩码、默认网关、DNS服务器等网络配置
- 地址复用：地址租约->自动回收不再使用的IP地址
###### 标准流程
1. DHCP 发现报文(**DHCP DISCOVER**)
	- 源：**0.0.0.0:68**(DHCP客户端为68号端口)
	- 目标：**255.255.255.255:67**(DHCP服务器为67号端口)
2. DHCP 提供报文(**DHCP OFFER**)
	- 源：DHCP服务器IP地址:67
	- 目标：255.255.255.255:68
	- 携带：DHCP服务器地址+配置信息
3. DHCP 请求报文(**DHCP REQUEST**)
	- 源：0.0.0.0:68
	- 目标：255.255.255.255:67
	- 携带：接收租约的IP地址、租期+相应的DHCP服务器地址
4. DHCP ACK报文(**DHCP ACK**)
	- 源：DHCP服务器IP地址:67
	- 目标：255.255.255.255:68
	- 携带：DHCP服务器地址+确认的配置信息
- 广播：
	- 客户端始终没有IP地址，无法建立TCP连接(无法单播)
	- 使用UDP广播，DHCP协议保障可靠性
![[DHCP流程.PNG|303]]
###### 地址续租流程
- 租期过半时，客户端向原DHCP服务器发送**DHCP REQUEST**申请续租
- 服务器同意->**DHCP ACK**，延长租期
- 服务器拒绝->**DHCP NACK**，客户端需停止使用该IP，重新进行标准流程
###### 跨网段DHCP：中继代理
- 问题：路由器默认不转发广播包，若**服务器和客户端不在同一网段**上，客户端无法发现服务器
- 解决：部署DHCP中继代理(通常在路由器上)
	- 中继代理收到客户端的请求，以**单播**形式转发给远程DHCP服务器
	- 服务器回复给中继代理，再由中继代理**广播**给客户端
- 优势：跨网段统一IP分配管理，无需为每个网段单独部署DHCP服务器


#### 4.NAT 网络地址转换
###### 背景
- IPv4地址枯竭
###### 传统NAT
- 原理：在NAT路由器上，将`私网IP<->公网IP`做**一对一映射**
- 局限：有多少台私网设备，就需要多少个公网IP，地址复用效率极低
###### NAPT
- **原理**：
	- **同时转换IP地址+端口号**
	- 多个私网设备共享一个公网IP，通过**不同的映射端口号来区分**各自的会话
	- NAT路由器维护一张**转换表**，记录`私网IP:端口<->公网IP:端口`的对应关系
![[NAPT.PNG|586]]
- **转换表生命周期**：
	- 生成：TCP三次握手的SYN包发出时，自动创建表项
	- 删除：TCP四次挥手的FIN/ACK包处理后，表项被回收
- **优点**：
	- 极大缓解IPv4地址枯竭，多设备共享公网IP
	- 隐藏内部网络结构，提升安全性
	- 实现简单，部署广泛
- **缺点**：
	- **外部无法主动发起连接到内部设备**
	- 转换表操作带来性能开销
	- NAT路由器重启时会清空转换表，导致所有TCP连接中断
- **解决方案**
	- 改用IPv6：为每台设备分配独立公网IP，从根本上消除地址转换需求(待普及...)
	- **NAT穿透技术**：应用程序主动发现自己位于NAT设备之后+**主动获取NAT设备的公网IP**+主动建立端口映射条目->使用该映射条目对外通信，外网设备也能主动连接内网设备了


#### 5.ICMP 互联网控制报文协议
###### 定位
- Internet Control Message Protocol（互联网控制报文协议）
- 依附于IP层的**网络层辅助协议**，本身不传输业务数据，专门用于**网络控制与故障诊断**
###### 作用
- 可达性确认：验证IP包是否成功送达目标地址(ping)
- 错误报告：当IP包被丢弃时，告知源主机失败原因
- 网络优化：提供路由重定向等信息，辅助优化网络传输路径
###### 报文类型

| 类型   | 类型码 | 名称                  | 作用                             | 类别   |
| ---- | --- | ------------------- | ------------------------------ | ---- |
| 查询报文 | 8   | 回送请求 (Echo Request) | 主动探测目标是否可达（`ping` 发送）          | 诊断类  |
| 查询报文 | 0   | 回送应答 (Echo Reply)   | 对回送请求的响应（`ping` 接收）            | 诊断类  |
| 差错报文 | 3   | 目标不可达               | 目标主机 / 端口 / 网络不存在              | 错误通知 |
| 差错报文 | 4   | 原点抑制                | 通知源主机降低发送速率（拥塞控制，现已废弃）         | 错误通知 |
| 差错报文 | 5   | 重定向 / 改变路由          | 告知源主机更优的路由路径                   | 路径优化 |
| 差错报文 | 11  | 超时                  | TTL 耗尽或分片重组超时（`traceroute` 依赖） | 错误通知 |

###### 特性
- 封装方式：ICMP报文直接封装在IP数据包中传输
- **不可靠性**：ICMP本身只提供尽力而为的送达服务，并不保证送达
- 安全限制：部分网络禁用ICMP以防止攻击
###### 应用场景
- `ping`命令：发送Echo Request，接收Echo Reply，测试网络连通性和延迟
- `traceroute`命令：利用超时报文，逐跳追踪数据包传输路径

#### 6.IGMP 因特网组管理协议
###### 定位
- **与ICMP无关**
- 作用范围：
	- **主机(组播成员)与最后一跳路由器之间**，专门管理D类组播地址的成员关系
	- TTL=1，不跨网段传播(跨网段---PIM协议)
- 封装：IGMP报文封装在IP包内，协议号为`2`

![[IGMP作用范围.PNG|341]]
###### 工作机制(以IGMPv2为例)
- **常规查询与响应**：
	- 路由器周期性向**224.0.0.1**(**同一网段的所有主机和路由器**)发送查询，询问网段内的组播成员
	- 主机收到后启动**随机延迟计时器**(0~10秒)，超时后发送成员报告
	- **报告抑制机制**：若在超时前收到同组其他主机的报告，自己不再发送，减少冗余报文
	- 路由器只要收到任一主机的报告，就标记该组在本网段有成员，后续转发组播包到该网段 
- **离开组播组**：
	- 主机向**224.0.0.2**(**同一网段的所有路由器**)发送离开报文
	- 路由器收到后，连续发送2次特定组查询，确认本网段内是否还有该组成员
		- 若有主机响应->继续保留该组
		- 若无响应->从IGMP表删除该组，不再转发
###### 核心设计思想
1. **IGMP表只维护网段级组播组**
	- 路由器不记录~~每个组播组对应的**具体网卡IP**~~
	- 只记录：**本网段内是否存在该组播组的成员**
2. **组播包转发：网段广播，主机过滤**
	- 路由器收到组播包后，若该组在本网段有成员，就向整个网段**广播**这个组播包
	- 主机收到后：
		- 已加入该组->接收并处理
		- 未加入该组->直接丢弃
