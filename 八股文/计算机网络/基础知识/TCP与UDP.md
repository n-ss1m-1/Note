# TCP基本认识
#### 什么是TCP
TCP是**面向连接的**、**可靠的**、**基于字节流的**传输层通信协议
- 面向连接：一对一连接，不能像UDP一对多
- 可靠：无差错、无丢失、无重复、按序到达
- 基于字节流：不维护消息边界，但保证有序+去重

#### 什么是TCP连接
定义：为了保证可靠性和流量控制而维护的某些状态信息，这些信息的组合(包括 Socket、序列号和窗口大小)称为连接。
- **Socket**：包含IP地址和端口号
- **序列号**：解决乱序问题
- **窗口大小**：用于流量控制

#### 如何唯一确定一个TCP连接
由TCP四元组唯一确定一个连接：
- **源IP地址**
- **目标IP地址**
- **源端口号**
- **目标端口号**

## TCP与UDP的区别与联系
#### UDP头部格式

![[UDP头部格式.PNG|408]]
- 源端口号+目标端口号：告诉UDP协议应该发给哪个进程
- 包长度：UDP头部+数据的长度之和
- 校验和：检测是否有数据错误

#### TCP头部格式
![[TCP头部格式.PNG|463]]
- 源端口号+目标端口号：标识通信的本端进程和远端进程
- **序列号(SEQ)**：该报文段首字节的字节流编号 =》解决网络包 重复and乱序 问题
- **确认应答号(ACK)**：收到的最后一个字节的序列号+1，表示期望接收的下一字节编号 =》解决丢包问题
- 首部长度：标识TCP头部的字节数(存多少个4字节)，TCP头部最长60字节
- 窗口大小：表示本端TCP缓冲区大小 =》流量控制
- **标志字段**：
	- ACK：确认收到数据
	- SYN：用于建立连接(客户端发起连接时置1)
	- FIN：请求关闭连接(本端半关闭)
	- RST：重置连接标注，强制断开连接或拒绝连接(发生连接错误)
- 校验和：检查报文段数据是否出错
- 选项：
	- MSS：TCP报文的数据部分最大长度( = MTU - IP头部 - TCP头部)
	- SACK(Selective ACK)：选择性确认，只重传丢的那一段，不重传后面的所有
	- Timestamp 时间戳：用于更精确计算RTT+防止序列号回绕(PAWS)
	- Window Scale 窗口扩大因子：让TCP滑动窗口突破65535字节限制

#### 核心区别
1. **连接**
	- TCP：面向连接，传输数据之前需要建立连接
	- UDP：无连接，即刻传输数据
2. **服务对象**
	- TCP：一对一的两点服务(不支持[[IP#广播|广播]]，只支持单播)
	- UDP：一对一、一对多、多对多的交互通信(支持广播->一对多)
3. **可靠性**
	- TCP：可靠的数据传输，无差错、无丢失、无重复、按序到达
	- UDP：尽最大努力但不保证可靠地交付数据(但是可以基于UDP实现一个可靠的传输协议)
4. **拥塞控制、流量控制**
	- TCP：有
	- UDP：无，即使拥塞也不影响其发送速率
5. **首部开销**
	- TCP：较长，有一定开销，没有使用 选项 字段时固定20字节，使用则为变长的
	- UDP：开销较小，固定8字节
6. **传输方式**
	- TCP：流式传输，没有边界，但会保证可靠性和顺序
	- UDP：一个包为单位的传输，有边界，但可能丢包和乱序
7. **分片方式**
	- TCP：按MSS分段，传输层分段，传输层组装，丢哪段传哪段
	- UDP：按MTU分片，IP层分片，IP层组装，一个分片丢失就将整个UDP包丢弃，无重传
	- MSS = MTU - IP头部 - TCP头部

#### 不同的应用场景
- TCP：FTP文件传输、HTTP/HTTPS、STML邮件传输
- UDP：
	- 包总量较少的通信，如DNS、DHCP、SNMP等
	- 视频、音频等多媒体通信
	- 广播通信

#### TCP和UDP可以使用同一个端口吗
答案：**可以的**。
- 在数据链路层中，通过 **MAC 地址**来寻找局域⽹中的主机。
- 在网络层中，通过 **IP 地址**来寻找⽹络中互连的主机或路由器。
- 在传输层中，需要通过**端⼝号**进⾏寻址，来识别同⼀计算机中同时通信的不同应⽤程序。

- 所以，传输层的「端⼝号」的作⽤，是为了区分同⼀个主机上不同应⽤程序的数据包。
- 传输层有两个传输协议分别是 TCP 和 UDP，**在内核中是两个完全独⽴的软件模块**。

- 当主机收到数据包后，可以在 IP 包头的「协议号」字段知道该数据包是 TCP/UDP，所以可以根据这个信息确定送给哪个模块（TCP/UDP）处理，送给 TCP/UDP 模块的报⽂根据「端⼝号」确定送给哪个应⽤程序处理。

# TCP三次握手
#### 过程与图解
##### 图解
![[TCP三次握手.PNG|426]]
##### 过程
1. **准备阶段**
	- 客户端：状态为CLOSE
	- 服务器：状态为CLOSE，调用listen()后变为LISTEN
2. **第一次握手**：客户端发送SYN
	- 动作：客户端主动发起连接，向服务器发送一个SYN报文
	- 报文内容：
		- 标志位：SYN = 1(请求建立连接)
		- 序列号(SEQ)：client_isn(客户端随机生成的初始序列号)
	- 状态变化：客户端变为SYN_SENT(等待服务器应答)
	- 注意：**该报文不能携带应用数据**
3. **第二次握手**：服务器发送SYN+ACK
	- 动作：服务器收到客户端发送的SYN报文，确认客户端的连接请求，并向客户端发送SYN+ACK报文
	- 报文内容：
		- 标志位：SYN = 1(自己也需要建立连接)  +  ACK = 1(确认收到)
		- 确认号(ACK)：client_isn + 1(期望下次收到的序列号)
		- 序列号(SEQ)：sever_isn(服务器随机生成的初始序列号)
	- 状态变化：服务器发送完毕后，状态变为SYN_RCVD(等待客户端确认)
	- 注意：**该报文不能携带应用数据**
4. **第三次握手**：客户端发送ACK
	- 动作：客户端收到服务器发送的SYN+ACK报文，然后向服务器发送ACK报文
	- 报文内容：
		- 标志位：ACK = 1(确认收到)
		- 确认号(ACK)：ACK = server_isn +1(期望下次收到的序列号)
		- 序列号(SEQ)：SEQ = client_isn + 1
	- 状态变化：
		- 客户端发送ACK后，直接进入ESTABLISHED状态(连接建立完毕)
		- 服务器收到ACK后，也进入ESTABLISHED状态
	- 注意：**该报文可以携带应用数据**

#### 为什么是三次握手？不是两次、四次？
1. **防止历史连接的错误初始化**(主要原因)
	- 网络延迟导致SYN阻塞/服务器ACK丢失：旧的SYN先到达服务器
	- 如果是两次握手：服务器收到 SYN 就直接建立连接并分配资源，无法判断这是历史连接，导致连接错误和资源浪费
2. **可靠同步双方的初始序列号**
	- 三次握手：
		- 客户端SYN -> 服务器SYN+ACK(服务器确认客户端的ISN + 发送自己的ISN)
		- 客户端ACK(客户端确认服务器的ISN)
	- 两次握手：只能保证一方的 ISN 被确认，无法保证双方 ISN 可靠同步
	- 四次握手：(服务器SYN和ACK分成两次) 冗余，三次已足够
3. **避免资源浪费**
	- 两次握手下，服务器接收到的每个ACK都要建立连接，重复的错误 SYN 会导致多个冗余连接，浪费服务器资源。
	- 三次握手下，服务器只在收到最后一个ACK后才真正完成连接，避免重复资源分配。

#### 第一次握手丢失，会发生什么？
1. 客户端想和服务器建立连接：发送SYN报文，进入SYN_SENT状态
2. 客户端**超时重传**：**重传的SYN报文的序列号是和之前一样的**
3. 注：超时时间每次*2
![[第一次握手丢失.png|431]]

#### 第二次握手丢失，会发生什么？
1. 当服务端收到客户端的第⼀次握手后，就会回 SYN-ACK 报文给客户端，这个就是第⼆次握手，此时服务端会进⼊ SYN_RCVD 状态。
2. 第⼆次握手的 SYN-ACK 报⽂其实有两个⽬的 ：
	- 第⼆次握手里的 ACK， 是对第⼀次握手的确认报文；
	- 第⼆次握手里的 SYN，是服务端发起建立 TCP 连接的报文；
3. 如果第二次握手丢失：
	- 客户端：迟迟没有收到第二次握手->认为是第一次握手丢失->客户端超时重传SYN报文
	- 服务器：由于第二次握手丢失->服务器收不到第三次握手->服务器超时重传SYN-ACK报文
	![[第一次握手丢失.png|396]]

#### 第三次握手丢失，会发生什么?
1. 客户端收到服务器的SYN-ACK报文后，就会向服务器回复ACK报文，此时客户端进入ESTABLISHED状态
2. 两种情况：
	- **纯 ACK 丢失**：不重传，靠服务端重传 SYN+ACK 来恢复
	-  **带数据的 ACK 丢失**：会重传整个数据段，顺带完成第三次握手


#### 为什么每次建立连接的初始化序列号要求不一样？
1. 为了防止**历史报文**被下一个相同四元组的连接接收(主要)
	- 并非完全避免历史报文，也可能会发生序列号回绕问题
	- -> 通过选项中的Timestamp时间戳判断历史报文 
2. 安全性保障，防止黑客伪装相同序列号的TCP报文
3. 随机产生初始化序列号ISN的方法；基于时钟计时器

#### 既然IP层会分片，为什么TCP层还需要MSS呢？
**一句话总结**：
- **TCP使用MSS是为了在发送前就主动把数据分段，强制每个TCP报文段不超过MTU，让IP层不必分片。**
- **这样可以避免因单个IP分片丢失而导致的整个TCP报文的被迫全部重传，从而保障传输效率。**
**概念**：
- MTU：一个链路层帧所能承载的有效数据的最大长度，以太网中一般为1500字节
- MSS：除去IP和TCP头部之后，一个TCP报文段所能容纳的应用数据的最大长度
- MSS = MTU - IP首部 - TCP首部
**IP层分片的弊端**；
- TCP发送大于MTU的数据时，IP层被迫分片
- **一个分片丢失**，整个IP数据包无法重组
- TCP超时重传，只能重传整个TCP报文(**重传所有分片**)
TCP使用MSS的优势：
- 自动拆分：TCP 层直接把应用数据切成小于 MSS 的小块
- IP层无需分片：整体长度 < MTU
- **精准重传**：TCP 只需要重传**丢失的那一个报文段**

#### 什么是SYN攻击？如何避免？
**定义**：
- 攻击者短时间伪造不同 IP 地址的 SYN 报文，服务端每接收到⼀个 SYN 报文，就进⼊ SYN_RCVD 状态，但服务端发送出去的 ACK +SYN 报文，无法得到未知 IP 主机的 ACK 应答，久而久之就会**占满服务端的半连接队列**，此时服务器将会把接收到的所有SYN都丢弃，使得服务端不能为正常用户服务。
- **半连接队列**(SYN队列)
- **全连接队列**(Accept队列)
![[Linux内核的SYN队列和Accept队列.PNG|328]]
**应对方法**：
1. 增大netdev_max_backlog
2. 增大TCP半连接队列：
	- 增大 net.ipv4.tcp_max_syn_backlog
	- 增大 listen() 函数中的 backlog
	- 增大 net.core.somaxconn
3. 开启tcp_syncookies：
	- 不丢弃SYN包
	- 根据算法计算cookie值，作为第二次握手报文的序列号，然后返回给客户端
4. 减少SYN+ACK重传次数


# TCP四次挥手
#### 过程与图解
##### 图解
![[TCP四次挥手.PNG|376]]

##### 过程
1. 准备阶段
	- 客户端：**ESTABLISHED状态**(连接已建立，正在通信)
	- 服务器：**ESTABLISHED状态**
2. 第一次挥手
	- 动作：客户端**主动关闭连接**，向服务器发送FIN报文，表示**本端不再发送数据**
	- 报文内容：
		- 标志位：**FIN = 1**(请求关闭连接)
		- 序列号(SEQ)：client_seq
	- 状态变化：客户端->**FIN_WAIT1**(等待服务器对FIN的确认)
	- 注意：**该报文可以携带最后一段应用数据**
3. 第二次挥手
	- 动作：服务器收到FIN报文，向客户端发送ACK报文
	- 报文内容：
		- 标志位：**ACK = 1**
		- 确认号(ACK)：client_seq+1(期望下次收到的序列号)
		- 序列号(SEQ)：server_seq(当前待发送的下一字节序号)
	- 状态变化：
		- 服务器->CLOSE_WAIT(等待应用层处理完剩余数据，准备关闭连接)
		- 客户端收到ACK后->FIN_WAIT2
	- 注意：**服务器此时可能仍有未发送完的数据，会继续传输**
4. 第三次挥手
	- 动作：服务器处理完所有数据，向客户端发送FIN报文，表示**本端也不再发送数据**
	- 报文内容：
		- 标志位：**FIN = 1**
		- 确认号(ACK)：client_seq+1
		- 序列号(SEQ)：server_seq_new(处理完数据后新的序列号)
	- 状态变化：服务器->**LAST_ACK**(等待客户端最后的确认)
	- 注意：**该报文可以携带最后一段应用数据**
5. 第四次挥手
	- 动作：客户端收到服务器的FIN报文，回复ACK报文
	- 报文内容：
		- 标志位：**ACK = 1**
		- 确认号(ACK)：server_seq_new+1
		- 序列号(SEQ)：client_seq+1
	- 状态变化：
		- 客户端：发送ACK后->进入**TIME_WAIT**状态->等待**2MSL**(最长报文段寿命)->进入**CLOSE**状态
		- 服务器：收到ACK后->直接进入**CLOSE**状态
	- 注意：**只有主动关闭方会进入TIME_WAIT状态**


#### 为什么挥手需要四次？
- 主要是因为服务器的FIN和ACK分开发送：
	- 服务器先回复ACK是必须的，用于确认收到客户端的FIN
	- 但是此时服务器可能还有数据需要处理，所以**需要等到数据发送完毕后，才发送FIN报文**，表示本端也不再发送数据

#### 第一次挥手丢失，会发生什么？
1. 客户端发送FIN报文后，进入FIN_WAIT1状态
2. 但是客户端迟迟收不到ACK，触发**超时重传机制，重传FIN报文**
3. 重传次数达到tcp_orphan_retries后，不再发送FIN报文
4. 等待一段时间(上次超时时间的2倍)
5. 仍然未收到ACK
6. 直接进入CLOSE状态

#### 第二次挥手丢失，会发生什么？
1. 服务器收到客户端的FIN，回复ACK，此时服务器进入CLOSE_WAIT状态
2. 如果此时ACK丢失，**服务器不会重传ACK**
3. 由**客户端触发超时重传机制，重传FIN报文** (即1st重传机制)
4. 重传次数达到tcp_orphan_retries后，不再发送FIN报文，等待一段时间(上次超时时间的2倍)，如果仍然未收到ACK，直接进入CLOSE状态

####  第三次挥手丢失，会发生什么？
- 客户端视角：FIN_WAIT1--(收到服务器ACK后)->FIN_WAIT2
	- 如果客户端最初使用**close()关闭连接**：
		1. FIN_WAIT2状态最多等待tcp_fin_timeout时长(默认60s)
		2. 如果还是收不到FIN报文，就直接CLOSE状态
	- 如果客户端最初使用**shutdown()关闭连接**：
		1. FIN_WAIT2状态将一直等待
		2. 直到收到服务器的FIN报文
- 服务器视角：
	1. CLOSE_WAIT--(发送FIN报文)->LAST_ACK
	2. 此时服务器迟迟收不到ACK，触发**超时重传机制，重传FIN报文** (如1st重传机制)
	3. 重传次数达到tcp_orphan_retries后，不再发送FIN报文
	4. 等待一段时间(上次超时时间的2倍)
	5. 仍然未收到ACK
	6. 直接进入CLOSE状态
	
####  第四次挥手丢失，会发生什么？
1. 客户端收到服务器的FIN报文，回复ACK报文，此时客户端进入TIME_WAIT状态
2. TIME_WAIT会持续2MSL才会进入CLOSE状态
3. 服务器由于没有收到ACK，处于LAST_ACK状态
4. 服务器超时重传FIN报文，重传次数由tcp_orphan_retries控制
5. 重传过程中，客户端再次收到FIN报文，会重置定时器，重新等待2MSL
6. 重传超过次数，服务器就直接关闭，进入CLOSE状态
7. 客户端最后等待2MSL后，自然关闭，进入CLOSE状态

#### TIME_WAIT
##### 为什么需要TIME_WAIT状态？
1. 防止历史连接中的数据，被后面相同四元组的连接错误接收
2. 保证被动关闭的一方能接收到最后的ACK，从而帮助其正确的关闭

##### 为什么TIME_WAIT状态等待的时间是2MSL？
1. MSL(Maximum Segment Lifetime)，报文最大生存时间，超过该时间报文将被丢弃
2. 网络中可能存在来自发送方的数据包，这些数据包被接收方处理后又会响应发送方，这样的一来一回需要等待2MSL的时间
3. 1个MSL确保最后发送方的ACK能到达对端，1个MSL确保对端没有收到ACK重传的FIN报文可以到达
4. 2MSL是从客户端接收到FIN+发送ACK开始计时的。如果在TIME_WAIT期间，客户端又接收到服务器重发的FIN报文，则2MSL重新计时

##### TIME_WAIT过多的危害？
1. (服务器)系统资源占用：如文件描述符、内存、CPU、线程资源
2. (客户端)端口资源占用：一个TCP连接至少占用一个本地端口 / 但是一定条件下也可以重复使用(四元组任一元素不同，即可区分不同的TCP连接)

# Socket编程
#### 针对TCP的Socket编程-连接建立过程
- 服务器(被动接受连接):
	1. socket()：创建**监听socket**，初始状态**CLOSE**
	2. bind()：将监听socket绑定到固定的 IP + 端口 上
	3. listen()：将socket状态从CLOSE->**LISTEN**，内核开始维护半连接队列和全连接队列
	4. accept()：阻塞等待，从全连接队列中取出一个已经完成三次握手的连接，返回一个**新的已连接socket**，用于后续数据传输
	5. read() / write()：通过已连接socket进行数据读写
	6. close()：关闭已连接socket，触发四次挥手，释放资源
- 客户端(主动发起连接)：
	1. socket()：创建客户端socket
	2. connect()：向服务器的 IP + 端口 发起连接请求，触发TCP**三次握手**。握手成功后客户端进入**ESTABLISHED**状态
	3. read() / write()：与服务器进行数据读写
	4. close()：关闭客户端socket，触发四次挥手
![[Socket编程-连接建立的过程.PNG|457]]


#### listen()的参数backlog的意义？
- 函数原型：int listen(int socketfd, int backlog);
- 参数一：监听socket的文件描述符
- 参数二：在linux历史版本上有变化
	- 早期linux：backlog指SYN队列大小
	- linux内核2.2之后：backlog指accept队列大小
				  =》accept队列长度=min(backlog,somaxconn)


#### connect()和accept()发生在三次握手的哪一步？
1. 客户端connect()是在 第一次握手开始---第二次握手成功返回 
2. 服务器accept()是在 第二次握手开始---第三次握手成功返回 
![[Socket编程+TCP三次握手.PNG|424]]


#### 客户端调用close()之后的流程
1. 客户端调用close()，表明**客户端没有数据需要发送**，此时向服务器发送FIN报文，客户端进入FIN_WAIT1状态
2. 服务器接收到FIN报文，TCP协议栈会在服务器的接收缓冲区插入一个文件结束符**EOF**，然后服务器进入CLOSE_WAIT状态
3. 应用程序处理完前面的所有数据之后，通过 **read()返回 0** 感知到这个EOF，于是调用**close关闭已连接的socket**，这使得服务器也发出FIN包，进入LAST_ACK状态
4. 客户端接收到FIN包，回复ACK，客户端进入TIME_WAIT状态
5. 服务器收到ACK，进入CLOSE状态
6. 客户端等待2MSL之后，进入CLOSE状态
![[Socket编程+TCP四次挥手.PNG|465]]


#### 没有accept()可以建立连接吗？
**答案**：**可以的**
**原因**：
- **accept()系统调用不参与TCP三次握手过程**，它只负责从全连接队列中取出一个已经建立连接的socket，用户层通过accept拿到该socket之后，就可以对该socket进行读写操作了
- 如果不调用accept()，只会导致用户层无法拿到socket，也就**无法进行读写操作**而已，但是连接已经建立完成(已完成三次握手)
![[没有accept()可以建立连接.PNG|426]]


#### 没有listen()可以建立连接吗？
**答案**：**可以的**
**局限**：(**没有服务器参与**)
- TCP自连接：客户端自己与自己形成连接
- TCP同时打开：两个客户端同时向对方发出请求建立连接

# TCP的重传机制
#### 超时重传
###### 定义
- 重传机制的一种方式，即在发送数据时设置一个**定时器**，在超过指定时间后，仍未收到对方的ACK报文，则会重发数据。
###### 分类
- 数据包丢失
- 确认应答丢失
###### RTO怎么设置
- **RTT**(Round-Trip Time)：往返时延 = 接收到确认的时刻 - 数据发送时刻
- 超时重传时间(**RTO**，Retransmissin Timeout)
	- RTO较长：重传速度慢，效率低
	- RTO较短：可能没有丢失就重传，导致网络更加拥塞、更多超时、更多重传
- RTO：应该**略大于RTT**，且是**动态变化**的
###### 超时间隔加倍
- 每当遇到超时重传的时候，都会将下一次超时时间间隔设置为当前值的2倍
- 连续两次超时，说明网络环境差，不宜频繁反复发送
###### 不足
- 触发超时重传的问题是，超时周期可能相对较长哪个，有没有更快的方式？↓

#### 快速重传
###### 定义
- Fast Retransmit：不以时间为驱动，而是以数据驱动重传。
###### 工作机制
- 当收到**三个相同的ACK报文**时，会在定时器过期之前，重传丢失的报文段
![[快速重传.PNG|353]]
###### 不足
- 重传的时候，是重传一个，还是重传所有？↓
- 重传一个：效率低，可能是连续丢失好几个报文段
- 重传所有：浪费资源，可能只丢失了一个报文段

#### SACK 方法
###### 工作机制
- SACK(Selective Acknowledgment)：**选择性确认**
- 在TCP头部的 选项字段 添加SACK信息，告知发送方**已接收的数据块范围**，发送方就可以**只重传丢失的部分**，提升重传效率。
![[SACK.PNG|585]]
#### Duplicate SACK
###### 作用
- 使用SACK告知发送方，**有哪些数据被重复接收了**
- 让发送方知道，是发出去的包丢失了，还是接收方回应的ACK丢失了
- 可以知道是不是 发送方 的数据包被网络延迟了

# TCP的滑动窗口
#### 为什么要引入滑动窗口？
- 原始TCP是**停等协议**：
	- 工作机制：发一个包->等待ACK->再发下一个包
	- 不足：网络往返时间越长，吞吐量越低，效率低
- 滑动窗口：允许**连续发送**多个包，**无需逐个等待**确定，大幅提升网络利用率
#### 滑动窗口的本质
- 操作系统开辟的缓存空间
	- 用于暂存已发送但未确认的数据，收到ACK后清理
	- 用于暂存已收到并确认的数据，应用层读取后清理
#### 发送方的滑动窗口
- **组成**：
	1. 已发送+收到ACK：数据已确认，可从缓存清除
	2. 已发送+未收到ACK：等待接收方确认，暂存
	3. 未发送+在接收方处理范围内：接收方还有空间，可立即发送
	4. 未发生+不在接收方处理范围内：接收方窗口已满，暂停发送
- 发送窗口(`swnd`) = 已发送未确认(2) + 未发生且在接收方处理范围内(3)
- **可用窗口(3)** = 发送窗口大小 - 已发送未确认的数据量 , 可用窗口=0时 发送方必须停止发送
- **累计确认**：接收方可以用后一个ACK确认前面的所有数据->非最后一个ACK丢失不影响
![[累计确认.PNG|397]]

#### 接收方的滑动窗口
- **组成**：
	1. 已成功接收并确认：等待应用层读取
	2. 未收到但可以接收：接收缓冲区剩余空间(**接收窗口大小**)(`rwnd`)
	3. 未收到且不可以接收：缓冲区已满，需等待应用层读取后腾出空间
- **接收窗口大小**会随应用层读取速度动态变化，并**通过TCP报文(window字段)告知发送方**

#### 窗口大小由哪一方决定？
- 答案：**接收方**
- 发送方发送的数据大小不能超过接收窗口大小(2)，否则接收方就无法正常接收到数据
- 接收方通过**TCP头部的`window`字段告知**发送方，自己的接收窗口大小，从而平衡发送和接收速度
#### 发送窗口和接收窗口的大小是相等的吗？
- 不完全相等，**接收窗口大小 ≈ 发送窗口大小**
- 滑动窗口大小是变化的，接收方的应用程序读取过程中，接收窗口增大，同时通过TCP报文的`Window`字段告诉发送方，这个传播过程是有时延的，所以二者是约等于的关系。


# 流量控制
#### 原理
- 本质：**让发送方根据接收方的处理能力来控制发送速率**，避免接收方缓冲区溢出，防止数据丢失和网络资源浪费
- 基于**滑动窗口机制**实现，接收方通过TCP报文的`Window`字段动态告知发送方自己的接收窗口大小
#### 流程
1. **接收方维护接收窗口**
	- 接收窗口大小 = 接收缓冲区剩余可接收的数据量
	- 应用层读取数据后，窗口向右滑动，窗口大小动态调整
2. **通告窗口大小**
	 - 接收方在ACK报文中，将当前接收窗口大小(window字段)告知发送方
	 - 发送方控制发送窗口 <= 接收窗口，限制发送速率
3. **发送方控制发送速率**
	- 发送方收到ACK后，窗口向右滑动，释放已确认的数据缓存
	- 若接收窗口变小，则发送速率减慢；若接收窗口为0，则停止发送数据
#### 关键问题
###### 窗口关闭
- 问题：接收窗口为0时，发送方停止发送。若接收方恢复后**通告非零窗口的ACK报文丢失**，会导致双方**相互等待**，形成死锁。
- 解决：发送方收到零窗口通知后，启动持续计时器。超时后发送**窗口探测报文**，接收方必须回复当前窗口大小，从而打破死锁。
###### 糊涂窗口综合症
- 问题：(接收方繁忙)接收方频繁通告极小窗口，发送方频繁发送极小数据包，导致TCP/IP头部开销占比过高，网络利用率低。
- 解决：
	- 接收方：窗口大小<`min(MSS,缓冲区一半)`时，通告窗口大小为0，直到窗口足够大再开放
	- 发送方：使用**Nagle算法**，延迟发送小数据，直到`满足窗口>=MSS && 数据>=MSS` 或 `收到之前数据的ACK`才发送
- 关闭Nagle算法：
	`setsockopt(sock_fd, IPPROTO_TCP, TCP_NODELAY, (char *)&value,sizeof(int));`


# 拥塞控制
#### 定义
- 通过拥塞判断当前的网络状况，从而控制发送方的发送速率，防止因网络过载导致的时延、丢包、重传演变为恶性循环。
#### 对比
- 流量控制：避免发送方的数据填满 接收方缓冲区
- 拥塞控制：避免发送方的数据填满 整个网络
#### 核心概念
- **拥塞窗口(**`cwnd`)：发送方维护的状态变量，代表当前网络允许发送的数据量
- **实际发送窗口**：`swnd = min(接收窗口rwnd，拥塞窗口cwnd)`，同时受流量控制和拥塞控制的限制
- **慢启动门限**(`ssthresh`)：区分慢启动和拥塞避免阶段的阈值，默认为65535字节
- **拥塞判断**：发送方超时重传or收到3个重复ACK，即认为网络发生拥塞
#### 四大核心算法
###### 慢启动(Slow Start)
- 触发时机：连接刚建立时 or 超时重传后
- 规则：每收到1个新的ACK，`cwnd += 1`，窗口指**数级增长**(1 → 2 → 4 → 8...)
- 阶段切换：当`cwnd>=ssthresh`时，切换为拥塞避免算法
 ![[慢启动.PNG|402]]

###### 拥塞避免算法
- 规则：每收到1个新的ACK，`cwnd += 1/cwnd`，窗口**线性增长**
- 目标：在不引发拥塞的前提下，逐步探测网络的最大承受能力
![[拥塞避免.PNG|412]]

###### 拥塞发生
1. **超时重传**(网络严重阻塞)
	- `ssthresh = cwnd/2`
	- `cwnd = 1`(重置为初始化，可能不是1)
	- 重新进入**慢启动**阶段
![[超时重传.PNG|439]]

2. **快速重传**(收到3个重复ACK，网络轻度阻塞)
	- `ssthresh = cwnd/2`
	- `cwnd = ssthresh`
	- 进入**快速恢复**阶段
###### 快速恢复
- 规则：
	- `cwnd = ssthresh+3`(代表已有3个包被确认)
	- 重传丢失的包
	- 若再收到重复ACK，`cwnd +=1`(能收到重复ACK->网络未堵死+空间已腾出->快速重传)
	- 若**收到新的ACK**([[TCP与UDP#发送方的滑动窗口|累计确认]])，`cwnd = ssthresh`，切换回**拥塞避免**阶段(拥塞刚缓和不宜激进发包，降低cwnd并回到拥塞避免，继续探测)
- 特点：相比超时重传更温和，在轻度拥塞的情况下，保证较高的传输速率，平稳过渡到正常发包节奏。
![[快速恢复.PNG|487]]
