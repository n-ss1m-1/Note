索引

优点：降低查找成本+降低排序成本
缺点：占用空间+降低更新表的速度

==========================
为什么 InnoDB 存储引擎选择使用 B+tree 索引结构？
1.相对于二叉树，层级更少，搜索效率高；
2.对于 B-tree，无论是叶子节点还是非叶子节点，都会保存数据，这样导致一页中存储的键值减少，指针跟着减少，要同样保存大量数据，只能增加树的高度，导致性能降低

==========================
索引分类：
1.根据功能分类：
分类		核心含义				关键特点					关键字
主键索引	基于表主键创建的索引	表中唯一存在，默认自动创建	PRIMARY
唯一索引	保证字段值唯一性		一张表可创建多个			UNIQUE
常规索引	加速普通数据的查询定位	一张表可创建多个			无
全文索引	检索文本中的关键词		一张表可创建多个			FULLTEXT

2.根据InnoDB 存储结构分类：		
聚集索引：
	特点1：索引与数据行合二为一，索引叶子节点直接存储完整行数据		
	特点2：必须且仅有一个
	选取规则：主键 → 第一个唯一索引 → 自动生成隐藏的rowid
二级索引：
	特点1：索引与数据分离，索引叶子节点仅存储主键值(id)+索引自身的所有字段
	特点2：表中可存在多个
	回表查询：先通过二级索引(name)，定位到目标数据的主键值(id)
			=>再通过聚集索引(主键)，用主键值查找并获取完整的行数据

==========================
语法：
1. 创建索引
CREATE  [UNIQUE|FULLTEXT]  INDEX  index_name  ON table_name (index_col_name,...);
UNIQUE：创建唯一索引，保证列值不重复
FULLTEXT：创建全文索引，用于文本关键词检索
不写修饰词则为常规索引，用于加速普通查询
index_col_name：字段列表

2. 查看索引
SHOW INDEX FROM table_name;

3. 删除索引
DROP INDEX index_name ON table_name;

==========================
性能分析：
1.查看当前数据库insert、update、delete、select的访问频次：
	show global status like 'Com_______';
2.慢查询日志定位瓶颈SQL
3.profile详情分析
4.explain + 查询语句 -> 查看SQL语句的执行计划：
1）Id：执行顺序控制->id 不同值越大越先执行，id 相同则按从上到下顺序执行。
2）select_type：查询类型识别 -> 区分查询类型（简单查询、主查询、子查询、联合查询等），帮助理解 SQL 的执行结构。
3）▲type：连接性能关键 -> 是评估 SQL 性能的核心指标。
	性能排序为：NULL > system > const > eq_ref > ref > range > index > all。
3.1）const/eq_ref：高性能，走主键 / 唯一索引；
3.2）range：范围查询，走索引范围；
3.3）index/all：性能较差，全索引 / 全表扫描。
4）possible_key：可能用到的索引；
5）Key：实际使用的索引（NULL 表示未走索引）；
6）Key_len：索引使用字节数，越短说明索引越精简、性能越好。
7）rows：MySQL 预估需扫描的行数，值越小越好；
8）filtered：返回结果行占读取行的比例，值越大说明过滤效率越高，SQL 性能越好。


==========================
索引使用规则：
1.最左前缀法则：联合索引必须从索引声明的最左字段开始匹配(否则不生效)，连续无跳过(否则后半部分失效)，where中书写的条件顺序不影响
2.范围查询：联合索引中，出现范围查询（>，<，between），则索引声明的右侧的字段的索引失效（使用>=则可消除该影响）
3.在索引列上进行运算操作=》索引失效
4.使用字符串类型字段时(类型隐式转换)，不加引号=》索引失效
5.模糊查询：尾部模糊匹配=》索引有效 ||| 头部模糊匹配=》索引失效
6.or连接的条件：如果or前的条件中的字段有索引，而后面的条件中的字段没有索引，则涉及的索引都不会被使用
7.如果MySQL评估使用索引比全表扫描更慢，则不使用索引

==========================
SQL提示：
1.use index(idx_xxx) => 建议使用该索引
2.ignore index(idx_xxx) => 忽略该索引
3.force index(idx_xxx) => 强制使用该索引
注：置于"from  表名"之后

==========================
覆盖索引：
单个索引中包含查询展示的所有字段(select后)+查询条件的所有字段(where后) => 可以避免回表查询

==========================
前缀索引：
1.场景：
	当字段类型为字符串（varchar，text 等）时，有时候需要索引很长的字符串，这会让索引变得很大，查询时，浪费大量的磁盘 IO，影响查询效率。此时可以只将字符串的一部分前缀，建立索引，这样可以大大节约索引空间，从而提高索引效率。

2.语法：
	create index idx_xxx on table_name(column(n));	#n即为前缀长度

3.选择性 = count(distinct 前缀字段) / count(*)	  #越接近1 速度越快 占用空间越大
	例：select count(distinct substring(email,1,5))/count(*) from tb_user;

==========================
索引设计原则：
1.针对于数据量较大，且查询比较频繁的表建立索引。
2.针对于常作为查询条件（where）、排序（order by）、分组（group by）操作的字段建立索引。
3.尽量选择区分度高的列作为索引，尽量建立唯一索引，区分度越高，使用索引的效率越高。
4.如果是字符串类型的字段，字段的长度较长，可以针对于字段的特点，建立前缀索引。
5.尽量使用联合索引，减少单列索引，查询时，联合索引很多时候可以覆盖索引，节省存储空间，避免回表，提高查询效率。
6.要控制索引的数量，索引并不是多多益善，索引越多，维护索引结构的代价也就越大，会影响增删改的效率。
7.如果索引列不能存储 NULL 值，请在创建表时使用 NOT NULL 约束它。当优化器知道每列是否包含 NULL 值时，它可以更好地确定哪个索引最有效地用于查询。
