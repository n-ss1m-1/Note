锁

==========================
全局锁：
定义：全局锁是对整个数据库实例加锁，加锁后数据库进入只读模式，所有写操作（DML/DDL/ 事务提交）都会被阻塞。
典型场景：全库逻辑备份（mysqldump），保证备份期间数据一致，无中途变更。

操作命令(命令行 | 先连接 mysql -uroot -p)：
1.加全局读锁：FLUSH TABLES WITH READ LOCK;	
2.全库备份：mysqldump -u用户 -p密码 库名 > 备份文件.sql
3.释放锁：UNLOCK TABLES;	
4.无锁备份(InnoDB MVCC)：mysqldump --single-transaction ...	

局限:
1.主库备份：备份期间业务无法写入，业务停摆，仅适合低峰期或非核心业务。
2.从库备份：阻塞 binlog 同步，导致主从延迟，影响高可用。


==========================
表级锁：
定义：表级锁，每次操作锁住整张表。锁定粒度大，发生锁冲突的概率最高，并发度最低。应用在 MyISAM、InnoDB、BDB 等存储引擎中。

分类方式1：
	1.表锁（与表级锁不同）
	2.元数据锁（meta data lock, MDL）
	3.意向锁
分类方式2：
	1.表共享读锁（read lock）
	2.表独占写锁（write lock）


==========================
(读写)表锁：
1.加锁：lock tables  表名..  read/write;	
2.释放锁：unlock tables;  /  客户端断开连接。
3.读共享，写独占：读锁不会阻塞其他客户端的读，但是会阻塞写。写锁既会阻塞其他客户端的读，又会阻塞其他客户端的写。


==========================
元数据锁(MDL)：
1.作用：保护表结构（元数据），避免 DML 与 DDL 冲突
2.加锁时机：访问表(开启事务)时【自动加锁】，无需手动操作  |  事务提交【自动解锁】
3.分类：
	读锁（共享）：DML（增删改查）时加 → 允许读写，阻塞 DDL
	写锁（排他）：DDL（ALTER TABLE）时加 → 阻塞所有读写
4.读共享，写独占：
	读锁之间兼容，与写锁互斥，写锁与所有锁互斥
5.查看当前的 MDL：
SELECT object_type,object_schema,object_name,lock_type,lock_duration 
FROM performance_schema.metadata_locks;

==========================
意向锁：
1. 作用机制：
	意向锁是 InnoDB 表级锁，专门用来【协调行锁与表锁】：
	当事务要给某行加行锁时，会先给整张表加对应的意向锁；
	后续如果有人要加表锁，只需检查意向锁，不用遍历全表看每行是否被锁，大幅提升效率。

2. 分类：
锁类型		英文缩写	触发语句
意向共享锁	IS		SELECT ...  LOCK IN SHARE MODE	
意向排他锁	IX		INSERT/UPDATE/DELETE、SELECT ...  FOR UPDATE	
注：意向锁之间（IS 与 IX、IS 与 IS、IX 与 IX）互不冲突阻塞，它们只是 预告 表内有行级锁。

3. 兼容性：
	加表读锁（read）：只要表上没有 IX 锁，就可以加。
	加表写锁（write）：只要表上有任何 IS/IX 锁，都必须等待。

4. 查看意向锁和行锁的信息：
SELECT object_schema, object_name, index_name, lock_type, lock_mode, lock_data
FROM performance_schema.data_locks;


==========================
行级锁：
特征：
	1.锁粒度：行级锁是 MySQL 中粒度最小的锁，仅锁定操作涉及的行数据。
	2.并发与冲突：锁冲突概率最低，并发度最高，是 InnoDB 支撑高并发的核心。
	3.实现依赖：行锁基于【索引】实现，若 SQL 未命中索引，会退化为表锁。
	4.支持引擎：仅 【InnoDB】 支持。

分类：
1.行锁(Record Lock)：锁定单个行记录的锁，防止其他事务对此行进行 update 和 delete。在 RC、RR 隔离级别下都支持。
2.间隙锁(Gap Lock)：锁定索引记录间隙（不含该记录），确保索引记录间隙不变，防止其他事务在这个间隙进行 insert，产生幻读。在 RR 隔离级别下支持。
3.临键锁(Next-Key Lock)：行锁和间隙锁组合，同时锁住数据，并锁住数据前面的间隙 Gap。在 RR 隔离级别下支持。

==========================
行锁：
行锁分类(读共享，写独占)：
	1.共享锁（S）：允许该事务去读一行，阻止其他事务获得相同数据集的排他锁。
	2.排他锁（X）：允许该事务更新数据，阻止其他事务获得相同数据集的共享锁和排他锁。

行锁与 SQL 对应：
SQL语句					行锁类型		说明
INSERT ...					排他锁		自动加锁
UPDATE ...				排他锁		自动加锁
DELETE ...					排他锁		自动加锁
SELECT(正常)				不加任何锁	普通查询不加锁（MVCC 实现一致性读）
SELECT ... 	LOCK IN SHARE MODE	共享锁	手动加共享锁
SELECT ... 	FOR UPDATE		排他锁		手动加排他锁

补充：？
1.默认情况下，InnoDB 在 REPEATABLE READ 事务隔离级别运行，InnoDB 使用 临键锁 进行搜索和索引扫描，以防止幻读，区间为(上一条, 当前条]。
2.针对唯一索引进行检索时，对已存在的记录进行等值匹配(不包括范围查询)时(不可能幻读)，将会自动优化为行锁(而非临键锁)。
3.InnoDB 的行锁是【针对索引】加的锁，不通过索引条件检索数据，那么 InnoDB 将对表中的所有记录加锁，此时就会【升级为表锁】。

查看意向锁及行锁的加锁情况：
select object_schema,object_name,index_name,lock_type,lock_mode,lock_data from performance_schema.data_locks;


==========================
临键锁与间隙锁：

1.范围查询(任何索引)：一律 临键锁
2.等值查询：
	唯一索引 + 等值 + 记录存在	行锁	
	唯一索引 + 等值 + 记录不存在	间隙锁	
	普通索引 + 等值 + 记录存在	临键锁(前面) or 退化间隙锁(查到最后一个满足条件的下一条)
3.间隙锁：只防插入，可共存，不阻塞其他间隙锁。
